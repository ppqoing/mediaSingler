# 重复组详情界面图片与视频 dHash 显示修改计划

> 日期：2026-07-16  
> 状态：已完成修改；Debug/Release x64 构建及测试通过

## 1. 修改目标

1. 在重复组详情界面中明确显示每个图片、视频成员的 dHash。
2. 图片显示一个图片 dHash；视频显示六个采样帧 dHash，并标明帧序号。
3. dHash 使用十六进制格式展示，支持鼠标选中和复制。
4. 区分有效值、未计算值和无效值，不能使用默认值冒充有效结果。
5. 旧版报告不包含媒体类型和 dHash 时，不再静默显示为空，而是明确提示用户重新生成报告。

## 2. 当前代码状态与问题

### 2.1 已有数据能力

`DedupCore/models/CoreModels.h` 中的 `DuplicateMember` 已包含：

- `image_dhash`：图片 dHash。
- `video_dhashes`：视频六帧 dHash。
- `has_video_dhashes`：视频 dHash 是否已经生成。
- `media_kind`：用于区分图片、视频及其他文件。

`DedupCore/dedup/DuplicateReportService.cpp` 当前使用第 2 版重复组编码，已经能够把上述字段写入并读出重复报告。

### 2.2 已有界面能力

`VideoScGUI/VideoScApp.cpp` 当前在“哈希值”单元格内混合显示 SHA-512 和 dHash：

- 图片显示一个 dHash。
- 视频逐行显示六帧 dHash。
- 缺失值显示“无（未计算）”。
- 值为 `0` 时显示“无效”。

但 SHA-512 与 dHash 共用一个较窄的列，六帧视频 dHash 容易被压缩或不易发现，不能形成清晰、稳定的独立展示区域。

### 2.3 旧报告兼容问题

重复报告读取仍兼容第 1 版编码，但第 1 版中没有 `media_kind`、图片 dHash 和视频 dHash。旧报告加载成功后：

- 成员类型保持为 `Other`。
- 图片和视频 dHash 均为空。
- 详情界面无法判断应显示图片 dHash 还是六帧视频 dHash。
- 用户只能看到 dHash 没有出现，但界面没有说明报告数据版本过旧。

由于旧报告本身没有保存这些数据，详情界面不能仅靠旧报告恢复 dHash。

## 3. 计划修改范围

### 3.1 重复组详情表格

修改 `VideoScGUI/VideoScApp.cpp` 的重复组详情表格：

1. 将当前“哈希值”列拆分为两个独立列：
   - `SHA-512`
   - `dHash`
2. SHA-512 列继续显示完整的 128 位十六进制字符串。
3. dHash 列按媒体类型显示：
   - 图片：显示一个 `0x` 开头、固定 16 位的十六进制 dHash。
   - 视频：显示“帧 1”至“帧 6”的六个 `0x` 开头、固定 16 位的十六进制 dHash。
   - 音频及其他文件：显示“不适用”。
4. 两列均保持单行值、禁止自动换行、支持鼠标调整列宽。
5. 所有哈希值继续使用只读可复制组件，避免普通文本难以完整复制。
6. 保持现有横向滚动能力，新增列不能压缩路径、预览等既有信息。

### 3.2 dHash 状态显示规则

统一使用以下状态：

| 媒体类型 | 数据状态 | 界面显示 |
| --- | --- | --- |
| 图片 | `image_dhash` 有值且非 `0` | 显示完整十六进制 dHash |
| 图片 | 没有 `image_dhash` | `无（未计算）` |
| 图片 | dHash 为 `0` | 显示十六进制值并标记 `无效` |
| 视频 | `has_video_dhashes == false` | `无（未计算）` |
| 视频 | 六帧均非 `0` | 显示六帧完整十六进制 dHash |
| 视频 | 任意一帧为 `0` | 仍显示六帧值；对应帧标记 `无效`，并提示该视频 dHash 整体无效 |
| 音频或其他 | 不参与视觉 dHash | `不适用` |

视频任意一帧为 `0` 时，继续遵守项目现有规则：该视频的整组视觉签名无效，不能作为有效 dHash 参与相似去重。

### 3.3 报告数据来源

1. 详情界面只读取 `DuplicateMember` 中已经随报告保存的 dHash，不在 UI 渲染期间重新计算媒体特征。
2. 不在每帧渲染期间访问 MySQL、RocksDB 或媒体文件，避免详情窗口滚动时产生数据库和磁盘读取卡顿。
3. 新生成的 SHA-512 精确报告和 dHash 相似报告都必须继续把媒体类型及 dHash 写入报告成员。
4. 不修改 SHA-512、图片 dHash、视频六帧 dHash 的计算算法和判定阈值。

### 3.4 旧报告处理

修改 `DedupCore/dedup/DuplicateReportService.cpp` 的报告解码逻辑：

1. 识别不包含媒体类型和 dHash 的第 1 版重复组记录。
2. 第 1 版报告不再作为可完整展示的报告静默加载。
3. 返回明确错误，例如：
   - `报告版本过旧，不包含媒体类型和 dHash，请重新生成重复报告。`
4. GUI 将该错误展示在重复报告窗口，不使用空值或 `Other` 类型伪装成正常结果。
5. 用户重新生成报告后继续使用当前第 2 版编码，无需修改 MySQL 和 RocksDB 中的媒体计算结果。

## 5. 实施步骤

1. 调整重复组详情表格列定义，把现有“哈希值”列拆分为 `SHA-512` 和 `dHash`。
2. 保留 SHA-512 的完整十六进制、只读和可复制行为。
3. 按 `media_kind` 分支渲染图片、视频、音频及其他文件的 dHash。
4. 为视频六帧值添加统一的帧序号、缺失状态和零值无效状态。
5. 增加视频“任意一帧为 `0` 则整体无效”的界面提示。
6. 调整列初始宽度及成员行高度，确保六帧视频 dHash 不被纵向裁剪。
7. 修改旧版报告解码错误信息，使旧报告缺少 dHash 时要求重新生成。
8. 补充报告持久化及兼容性测试。
9. 完成构建和测试后，人工检查图片组、视频组、精确重复组和相似重复组的详情显示。

## 6. 验收标准

1. 图片成员详情明确显示一个完整的图片 dHash。
2. 视频成员详情明确显示六个带帧序号的完整 dHash。
3. SHA-512 与 dHash 分列显示，内容不自动换行，列宽可由鼠标调整。
4. SHA-512、图片 dHash、视频六帧 dHash 均可完整复制。
5. 未计算、无效、不适用三种状态能够清晰区分。
6. 视频任意一帧 dHash 为 `0` 时，界面明确说明该视频 dHash 整体无效。
7. 加载不包含 dHash 的旧报告时，界面提示重新生成报告，不再静默缺失。
8. 新生成的 SHA-512 精确报告和 dHash 相似报告均能正确显示成员 dHash。
9. `DedupTests` 全部通过，Debug/Release x64 构建通过。

## 7. 不在本次范围

1. 不修改 dHash 计算算法、采样帧位置、汉明距离阈值或相似分组规则。
2. 不在详情界面增加手动重新计算 dHash 的按钮。
3. 不修改扫描、MySQL 同步、RocksDB 暂存和媒体分析流程。
4. 不修改重复组列表页的排序、虚拟加载或删除选择逻辑。
5. 不重新设计图片和视频缩略图预览区域。

## 8. 执行结果

1. 重复组详情表格已将原“哈希值”拆分为独立的 `SHA-512` 和 `dHash` 两列。

3. dHash 未计算时显示“无（未计算）”，图片或视频帧值为 `0` 时显示“无效”。
