# 多硬盘读取并发与 CPU 自适应计算线程改造计划

> 日期：2026-07-15  
> 状态：已完成实现；Debug/Release x64 构建及 DedupTests 32/32 已通过（2026-07-15）

## 1. 需求目标

1. 面向任意数量的物理硬盘自动创建读取通道，不再按 `PhysicalDriveN` 逐盘维护线程配置。
2. 配置页面增加文件读取并发总上限，以及单块 HDD、单块 SSD 对应的读取线程数。
3. 计算任务根据 CPU 使用率逐步增加并发，直到达到 CPU 目标负载或计算线程硬上限。
4. 扫描窗口显示实际读取并发、实际计算并发和 CPU 使用率。
5. 扫描任务窗口的“开始增量扫描”按钮改名为“开始扫描”，底层增量识别语义保持不变。

## 2. 当前实现结论

### 2.1 磁盘读取线程

- `StorageConfig::disk_read_threads` 保存 `PhysicalDriveN + 介质类型 + 线程数` 的逐盘覆盖表。
- `NativeFileDiscovery::PrepareRoot()` 已能通过 `QueryDiskTopology()` 自动获得物理存储键和 HDD/SSD 类型，多硬盘识别能力已经存在。
- `ScanCoordinator::ReadThreadsFor()` 优先查找逐盘覆盖，未命中时 HDD 使用 1 线程、SSD 使用 2 线程。
- `DiskHashScheduler` 为每个物理盘建立独立队列和固定数量工作线程，但目前没有“所有磁盘合计读取并发上限”。

因此本次不需要重新实现磁盘识别，主要是把“逐盘配置”替换为“按介质策略 + 全局并发上限”。

### 2.2 计算线程

- `compute.worker_threads` 当前是固定值，范围为 1～256。
- SHA-512 阶段使用固定容量 `ConcurrencyGate` 限制所有磁盘共享的哈希并发。
- 媒体阶段一次性创建 `worker_threads` 个线程，整个阶段不会动态增减。
- 项目内没有 CPU 使用率采样器；`ProgressSnapshot::active_compute_threads` 和 `queued_compute_tasks` 目前也没有完整赋值链路。

### 2.3 必须处理的耦合

当前单个哈希工作线程同时执行文件读取和 SHA-512 流式计算，读取与计算不是两个独立线程池。因此：

1. 单盘 HDD/SSD 线程数决定每块盘最多有多少个哈希任务可以并行。
2. 全局文件读取上限限制所有磁盘合计进行中的哈希任务数。
3. CPU 自适应计算并发是第三层限制，SHA-512 阶段的实际并发为上述三个限制共同作用后的结果。
4. 媒体阶段没有磁盘通道，只受 CPU 自适应计算并发和 FFmpeg 单任务线程数影响。

## 3. 推荐配置模型

### 3.1 磁盘读取配置

用以下字段替换逐盘数组：

| 字段 | 推荐默认值 | 含义 |
| --- | ---: | --- |
| `storage.max_concurrent_file_reads` | 8 | 所有物理盘合计正在读取文件的任务上限 |
| `storage.hdd_read_threads_per_disk` | 1 | 每块 HDD 最多并行读取文件数 |
| `storage.ssd_read_threads_per_disk` | 2 | 每块 SSD 最多并行读取文件数 |

未知介质推荐按 HDD 的保守值处理，避免无法识别的机械盘被高并发随机读取。

配置约束：

- 三个值均至少为 1。
- 单盘 HDD/SSD 值不能大于全局上限。
- 全局上限建议允许 1～256，单盘值建议允许 1～64。
- 内存提示改为按 `max_concurrent_file_reads × read_block_kib` 估算，不再累计手工磁盘表。

### 3.2 CPU 自适应计算配置

保留现有 `compute.worker_threads` 字段，并根据自动分配开关赋予两种语义：关闭自动分配时仍是原有固定计算线程数，开启时作为计算线程硬上限。新增：

| 字段 | 推荐默认值 | 含义 |
| --- | ---: | --- |
| `compute.adaptive_worker_threads` | `true` | 是否启用 CPU 自适应并发 |
| `compute.cpu_target_percent` | 90 | 达到该系统 CPU 使用率后停止增加 |

已确认增长策略：使用系统总 CPU；每个计算阶段从 1 个并发开始，每 1000 毫秒采样一次。存在排队任务且 CPU 低于页面配置目标时，每次只增加 1 个并发；达到目标或 `worker_threads` 硬上限后停止增加。本阶段只增加、不回收并发，SHA-512 切换到媒体特征阶段时重新从 1 开始评估。采样间隔和增长步长固定，不额外暴露到配置页面。

## 4. 计划修改

### 4.1 配置、持久化和兼容

修改：

- `DedupCore/config/AppConfig.h`
- `DedupCore/config/JsonConfigStore.cpp`
- `DedupCore/config/ConfigValidator.cpp`
- `DedupCore/orchestration/ScanOptionsCodec.cpp`

处理方式：

1. `StorageConfig` 改为按介质配置和全局并发上限。
2. `ComputeConfig` 增加自适应开关和 CPU 目标；现有 `worker_threads` 在自动模式下作为硬上限。
3. 新字段写入 `config.json` 和扫描检查点快照。
4. 新程序读取旧配置时忽略旧的 `disk_read_threads` 逐盘覆盖，并使用新的默认介质策略；保存后不再写入逐盘表。
5. 旧扫描快照恢复时不再保留逐盘线程语义，统一使用新的默认 HDD/SSD 策略和全局上限。

### 4.2 多盘读取调度

修改：

- `DedupCore/orchestration/ScanCoordinator.cpp`
- `DedupCore/scheduling/DiskHashScheduler.h/.cpp`

处理方式：

1. 删除 `ReadThreadsFor()` 的 `PhysicalDriveN` 查表逻辑，按运行时识别的 `StorageMediaType` 选择 HDD/SSD 单盘并发。
2. 继续为每个物理存储键保留独立队列、HDD 电梯排序和 SSD 顺序处理。
3. 在所有磁盘通道外增加全局文件读取并发闸门，确保实际同时读取数量不超过 `max_concurrent_file_reads`。
4. 全局闸门需要采用可取消等待，并在取消扫描时唤醒全部等待线程，不能重现“取消后进度不停止”。
5. 进度快照同时报告每盘配置并发、每盘活动并发和全局活动读取数。

### 4.3 CPU 采样与自适应并发

建议新增 `DedupCore/scheduling/CpuLoadSampler` 和自适应并发控制器：

1. 使用 Windows `GetSystemTimes()` 的两次差值计算系统总 CPU 使用率，不引入新第三方依赖。
2. 第一次采样只建立基线，不立刻扩容。
3. 只有存在待处理任务、未取消、CPU 低于目标且当前并发小于硬上限时才增加 1。
4. SHA-512 阶段把固定 `ConcurrencyGate` 改为可动态提高容量的闸门。
5. 媒体阶段按控制器允许的并发动态启用工作者，不能再固定创建 `worker_threads` 个忙工作者。
6. 自适应控制线程纳入现有异常边界、取消和 `join` 生命周期，任何采样失败都退化为最小并发并写诊断日志。

### 4.4 配置页面

修改 `VideoScGUI/VideoScApp.cpp/.h`：

1. 删除物理盘键、介质和逐盘线程表，以及“添加物理盘线程配置”相关编辑状态。
2. 在“SHA-512 与磁盘读取”区域增加：
   - 文件读取并发总上限。
   - 单块 HDD 读取线程数。
   - 单块 SSD 读取线程数。
   - 未知介质按 HDD 配置处理的说明。
3. 在“计算线程”区域增加“自动分配计算线程”复选框和系统 CPU 目标；原线程字段在自动模式显示为“计算线程硬上限”，关闭自动模式时显示为“固定计算线程数”。
4. 扫描窗口显示“当前计算并发 / 上限 / CPU 使用率 / 全局活动读取数”。
5. 把扫描任务窗口按钮文案从“开始增量扫描”改为“开始扫描”，继续调用现有 `StartScan()`，不改变增量复用和断点恢复逻辑。

## 5. 线程安全与异常处理要求

1. 自适应容量、活动计数和 CPU 快照使用原子值或受同一互斥锁保护。
2. 增加容量时必须唤醒等待工作者；取消时必须唤醒 CPU 控制线程、读取闸门和计算闸门。
3. 不允许在持有磁盘队列锁时等待全局闸门，避免多盘互相阻塞。
4. 所有新增线程均捕获 `std::bad_alloc`、`std::exception` 和未知异常，并进入现有 `ApplicationErrorLogger`。
5. 进度读取只复制快照，不让 GUI 直接访问调度器内部对象。

## 6. 测试计划

1. 旧配置包含逐盘数组时可以加载，新字段使用默认值，重新保存后不再产生逐盘配置。
2. 新配置和扫描快照保存/加载后字段一致，非法上下限被拒绝。
3. 模拟多块 HDD/SSD，验证每盘活动读取不超过介质配置、全局活动读取不超过总上限。
4. 全局上限小于磁盘工作线程总数时仍能公平推进所有磁盘，不发生永久饥饿。
5. CPU 低负载且有排队任务时并发逐步增加；达到目标或硬上限后停止增加。
6. CPU 采样失败时退化到安全并发，扫描仍能完成。
7. 自适应控制期间取消扫描，控制线程和全部工作线程均能及时退出。
8. SHA-512、媒体处理、断点恢复、DLL 异常边界和现有 32 个核心测试继续通过。
9. `Debug|x64`、`Release|x64` 全解决方案构建成功，`DedupTests` 全部通过。
10. 自动分配关闭时，SHA-512 和媒体阶段继续严格使用原固定 `worker_threads` 逻辑。
11. 扫描按钮显示为“开始扫描”，点击后仍进入原有增量扫描启动链路。

## 7. 已确认的关键语义

用户于 2026-07-15 确认：

1. 文件读取线程数量上限是所有硬盘合计的实际并发上限。
2. 自动分配计算线程由配置页面复选框控制；关闭时使用原有固定线程逻辑。
3. 自动模式使用系统总 CPU，CPU 目标由配置页面设置；每秒采样、每阶段从 1 开始、每次增加 1、阶段内只增不减。
4. 自适应同时作用于 SHA-512 和媒体特征阶段，阶段切换时重新评估。
5. 旧中断扫描允许使用新默认策略恢复，不要求保留旧逐盘线程语义。
