# 重复报告虚拟化、排序、媒体详情与执行日志改造计划

> 日期：2026-07-16  
> 状态：已执行完成；Debug/Release x64 构建通过，DedupTests 33/33 通过

## 1. 目标

1. 重复报告取消可见分页，全部结果表现为同一个连续滚动容器。
2. 主列表只加载和绘制当前位置附近的重复组，离开可见窗口的数据与纹理按上限淘汰，避免大报告卡顿。
3. 增加全报告排序：按每组可释放大小正序/倒序、按组内文件数量正序/倒序。
4. 图片详情使用保存到应用专属临时目录的缩略图；视频详情把 2×3 拼图按 UV 区域拆成六帧并排对比。
5. 配置页新增独立执行日志目录；常规执行轨迹与失败记录分别写入不同文件，并在每条记录后及时刷盘。
6. 详情界面以十六进制展示 SHA-512、图片 dHash 和视频六帧 dHash。
7. 每个文件完成 SHA-512 后，按摘要依次查询 RocksDB 和 MySQL 的媒体 dHash；存在当前算法版本的有效非默认值时直接复用，不重复计算。

## 2. 当前实现结论

### 2.1 重复报告列表

- `VideoScApp::LoadReportPage()` 固定调用 `DuplicateReportStore::LoadPage()`，默认每页 20 组。
- 主窗口持有完整的 `m_reportPageGroups`，每帧绘制当前页所有组、成员和选择状态。
- 详情窗口依赖当前页中的组对象，换页会关闭详情并清理纹理。
- “按条件选择当前页”和永久删除逻辑也直接依赖分页容器。

### 2.2 排序

- 报告目前只按生成时的 ordinal 顺序存储，没有轻量组摘要或全报告排序索引。
- `DuplicateGroup` 已有 `members.size()` 和 `reclaimable_bytes`；用户已确认大小排序统一采用每组 `reclaimable_bytes`。

### 2.3 图片和视频详情

- 图片当前直接从原文件经 WIC 解码并按 `image_preview_long_edge` 缩放成 GPU 纹理，没有保存临时缩略图文件；用户已要求先落到临时目录。
- 视频当前优先显示一张 2×3 拼图；同一张纹理可以使用 ImGui UV 坐标无复制地拆成六个画面。
- 视频拼图按既有设计不是扫描完整性的必要条件，因此历史数据或保存失败的数据可能只有六帧 dHash、没有拼图文件。
- 精确 SHA-512 报告的 `DuplicateMember` 当前没有保存 `media_kind`，详情窗口无法可靠区分精确重复图片与视频。

### 2.4 日志

- `ScanErrorLogger` 已把 SHA-512 打开失败、读取失败、超时、文件变化和计算失败写入 `scan-errors.jsonl`。
- 媒体分析失败当前只增加失败计数并写入 `media-error-v1`，没有记录路径、状态码、原生错误码和 DLL 错误文本。这正是界面显示 54 个计算失败、日志却为空的直接原因。
- `ApplicationErrorLogger` 负责异常边界；`OperationLogger` 负责永久删除审计；崩溃转储写入日志目录下的 `crash`。
- 配置页已经存在通用“日志目录”，还没有独立的“执行日志目录”。

### 2.5 SHA-512 后的媒体复用

- 当前 `HandleHashCompleted()` 已先按 SHA-512 查询 RocksDB；只有 RocksDB 完全没有该内容记录时才查询 MySQL。
- 如果 RocksDB 有内容记录但媒体 dHash 缺失、是默认值或算法版本不匹配，当前逻辑不会继续查询 MySQL，而会直接创建媒体计算任务。
- 当前 `MediaComplete()` 对图片只判断 `image_dhash.has_value()`，对视频只判断 `has_video_dhashes`，没有排除默认零值。
- 因此同一 SHA-512 的有效媒体结果只存在于 MySQL、而本地存在不完整占位记录时，仍可能发生不必要的重复计算。

## 3. 目标设计

### 3.1 单容器虚拟滚动

主窗口改为一个固定 ID 的 `ImGui::BeginChild()` 滚动容器，不再展示页码、上一页和下一页。

报告数据拆为两层：

1. **轻量摘要索引**：每组只保留 generation、ordinal、group_id、成员数量、排序大小等固定字段。
2. **完整组块缓存**：按当前位置成块读取 `DuplicateGroup`，只缓存可见区域前后有限数量的块。

列表以“组标题行 + 成员文本行”展平为固定高度虚拟行，使用 `ImGuiListClipper` 只提交可见行。滚动到尚未缓存的区间时，根据摘要的 ordinal 批量读取对应完整组；离开窗口较远的块按 LRU 淘汰。

约束：

- 拖动滚动条可以直接跳到报告任意位置，不能要求从头逐批滚动。
- 完整组和纹理缓存均有硬上限；摘要不包含路径字符串和媒体纹理。
- 当前详情组单独持有，不依赖主列表缓存是否被淘汰。
- 手工删除选择使用独立的 `group_id -> selected path_id` 状态保存，块淘汰后不能丢失。
- “按条件选择当前页”改为“按条件选择当前可见组”；“按条件选择全部报告”继续后台流式读取全部报告，不一次性加载完整组。

### 3.2 全报告排序

新增报告组摘要结构和读取接口，报告生成时与完整组原子保存摘要。加载报告时只读取摘要数组，然后按当前排序模式排列摘要，不解码全部路径和证据。

排序模式计划为：

- 默认顺序：报告生成顺序。
- 可释放大小正序。
- 可释放大小倒序。
- 组内文件数量正序。
- 组内文件数量倒序。

排序切换后：

1. 保留按 `group_id` 保存的手工选择。
2. 清空完整组块缓存和非详情纹理。
3. 重建轻量顺序与虚拟行前缀索引。
4. 滚动位置回到顶部。

旧报告没有摘要时，在第一次加载该报告代时流式补建摘要；不要求重新连接 MySQL 生成报告。

### 3.3 图片缩略图与视频六帧详情

`DuplicateMember` 增加稳定的 `media_kind`、可选图片 dHash、视频六帧 dHash及其有效标志；继续使用已有 `content_sha512`。报告二进制记录版本递增并兼容旧记录；精确报告 MySQL 查询补充 `media_kind`、`image_dhash` 和 `video_dhashes`，相似报告直接从 `ShaFileData` 填入。

图片详情：

- 在后台使用 WIC 按 `image_preview_long_edge` 生成有界缩略图文件，再从临时文件创建 GPU 纹理。
- 临时缩略图保存到 Windows 临时目录下的应用本次运行专属子目录，按内容 SHA-512 命名以复用相同图片内容。
- 正常关闭应用时先释放纹理，再递归删除本次运行的临时缩略图目录。
- 不把原始大图尺寸直接上传到显存，也不要求扫描阶段生成永久图片缩略图文件。
- 解码失败写执行日志，界面显示“缩略图不可用”。

视频详情：

- 对现有 2×3 拼图纹理使用六组 UV 坐标裁剪，显示为“帧 1～帧 6”，不创建六份重复纹理。
- 六帧对比与相似证据中的逐帧汉明距离顺序一致。
- 若历史视频没有拼图，打开详情时在后台按需生成拼图；不改变“有六帧 dHash 即视为媒体计算完整”的既有规则。
- 按需生成失败写执行日志，不阻塞其他成员详情。

十六进制字段：

- SHA-512 固定显示为 128 位小写十六进制字符，不带分隔符。
- 图片 dHash 固定显示为 `0x` 加 16 位小写十六进制字符。
- 视频按帧 1～帧 6 分别显示 `0x` 加 16 位小写十六进制字符，并与六帧画面和逐帧汉明距离保持同一顺序。
- 所有哈希文本使用可复制纯文本控件；旧报告缺少 dHash 时明确显示“旧报告未保存该值”，不显示伪造的零值。

### 3.4 SHA-512 索引复用媒体 dHash

图片或视频完成 SHA-512 后执行统一的内容能力解析：

1. 使用 SHA-512 十六进制键查询 RocksDB `ShaFileData`。
2. 本地记录的媒体类型、算法版本和 dHash 都有效且非默认值时直接复用。
3. 本地记录不存在或媒体能力无效时，继续使用同一 SHA-512 查询 MySQL `sha512_file_data`，不能因为本地有占位记录而跳过远端查询。
4. MySQL 返回有效媒体能力时，把完整内容记录回填 RocksDB，并取消该 SHA-512 的媒体计算任务。
5. MySQL 不可用时保留既有降级语义：仅根据本地有效结果决定；本地无有效结果才计算。
6. 本地和 MySQL 都没有当前算法版本的有效非默认 dHash 时，才创建媒体任务。
7. 多个路径拥有相同 SHA-512 时继续使用 `MediaTaskKey(scan_id, sha512)` 去重，同一内容最多计算一次。

有效性仍必须同时校验 `media_algorithm_version == options_.algorithm_version()`，避免复用旧算法 dHash。图片 dHash 为 `0` 时无效；视频六帧中任意一帧为 `0` 时整个媒体结果无效。

### 3.5 独立执行日志与失败日志

`LoggingConfig` 新增独立的 `execution_directory`，不复用现有异常/诊断日志目录。该目录至少维护两个独立滚动 JSONL 文件：

- `execution.jsonl`：常规任务执行轨迹。
- `execution-failures.jsonl`：读取、计算和任务执行失败记录。

两个文件都使用互斥串行追加；每条完整 JSON 写入后立即 `flush`，进程异常退出时尽量保留最后一条已提交记录。日志滚动大小、数量和保留天数暂时复用现有 `LoggingConfig` 参数，避免增加无必要的重复配置。

失败计数和日志采用同一提交规则：任何文件只有在 `execution-failures.jsonl` 成功写入并刷盘后才能增加 GUI 失败计数。失败日志自身无法写入时，不允许静默增加计数或继续伪装为已记录，而是进入现有应用异常系统并把扫描置为失败，确保界面失败数始终能追溯到日志记录。

失败记录至少包含：

- UTC 时间、任务/扫描 ID、阶段和操作类型。
- 文件完整路径、物理盘键、媒体类型。
- 稳定状态名、模块状态码、Win32/FFmpeg 原生错误码。
- 读取失败偏移和已读取字节数（适用时）。
- 安全错误文本；不记录 MySQL 密码、SQL 全文或其他密钥。

明确覆盖：

- 文件发现/清单读取失败。
- SHA-512 打开、读取、超时、文件变化、加密计算失败。
- 图片 dHash、视频六帧 dHash、媒体元数据和拼图生成失败。
- 详情页图片缩略图解码、历史视频拼图按需生成失败。
- MySQL、RocksDB、报告生成和永久删除执行失败。

原有 `application-errors.jsonl`、崩溃转储和永久删除审计日志继续保留，不与执行轨迹或失败记录合并。

## 5. 测试与验收

3. 四种排序作用于整个报告，不是只排序当前缓存块；相同键用 `group_id` 保证稳定次序。
4. 排序和缓存淘汰后，已选择删除的成员不丢失，同组仍至少保留一个文件。
5. 图片详情纹理长边不超过配置，缩略图文件位于本次运行临时目录；正常关闭后该目录不存在。
6. SHA-512、图片 dHash、视频六帧 dHash 均按固定宽度十六进制显示并可复制；旧报告缺失值时显示兼容提示。
7. 视频详情显示六个独立裁剪画面且顺序正确；没有拼图时后台补生成，失败不会卡住 UI，并生成包含路径和状态码的失败日志。
8. `execution.jsonl` 与 `execution-failures.jsonl` 分开滚动，SHA、媒体、缩略图各类失败均只进入失败文件；每条记录及时刷盘。
9. 构造 54 个媒体计算失败，GUI 失败计数必须为 54，`execution-failures.jsonl` 必须存在 54 条对应记录且每条包含路径、阶段和状态码。
10. 旧配置、旧报告代可以加载；旧报告摘要能按需补建。
11. RocksDB 只有不完整占位记录、MySQL 有同 SHA-512 有效 dHash 时，不创建媒体任务并把远端内容回填本地。
12. RocksDB 与 MySQL 都无有效 dHash 时只创建一个按 SHA-512 去重的媒体任务；MySQL 不可用时仍可本地计算。
13. 算法版本不匹配、图片 dHash 为 `0`、视频任意一帧 dHash 为 `0` 时不能复用并必须重新计算。
14. Debug/Release x64 全量构建通过，`DedupTests` 全部通过。

## 6. 已确认语义

1. 文件大小排序按每组 `reclaimable_bytes`。
2. 可释放大小和组内文件数量都提供正序、倒序。
3. 新增独立执行日志目录，常规执行日志和失败记录使用不同文件，并及时写入刷盘。
4. 图片缩略图保存到临时目录，关闭应用时删除。
5. 详情界面按固定宽度十六进制显示 SHA-512 和 dHash 相关值。
6. `execution.jsonl` 只记录任务开始/结束、阶段切换、同步批次、报告生成和删除批次摘要，不逐文件记录成功事件；`execution-failures.jsonl` 记录扫描、SHA、媒体、缩略图、MySQL、RocksDB、报告与删除的所有失败。
7. GUI 中的每个失败计数都必须有对应失败日志；失败日志写入失败时升级为任务失败。
8. 临时图片目录使用 `%TEMP%\\VideoScGUI\\preview-<run-id>`；正常关闭立即删除，异常退出遗留目录在下一次启动时清理。

## 7. dHash 默认值确认

用户已确认采用严格非零规则：

- 图片：`image_dhash` 缺失或等于 `0` 均视为无效，需要计算。
- 视频：`has_video_dhashes == false`，或六帧中任意一帧等于 `0`，均视为无效，需要重新计算全部六帧。
- 无论数值如何，`media_algorithm_version` 不匹配都视为无效。
- RocksDB 记录无效时仍继续查询 MySQL；只有本地和 MySQL 都无有效值时才进入媒体计算。

## 8. 执行结果

1. 报告存储新增轻量组摘要，并与完整组原子写入；旧报告首次加载时自动补建摘要。

3. 已提供生成顺序、可释放大小正序/倒序、组内文件数量正序/倒序五种全报告排序，并使用独立选择状态避免排序或缓存淘汰后丢失选择。
4. 图片详情生成本次运行专属 PNG 临时缩略图；视频拼图拆为六个 UV 区域。历史视频缺失拼图时在后台生成临时拼图，失败写入执行失败日志。
5. 详情已显示可复制的 128 位小写 SHA-512、图片 `0x` 加 16 位 dHash、视频六帧 `0x` 加 16 位 dHash。
6. 新增独立 `execution.jsonl` 与 `execution-failures.jsonl`，扫描阶段、同步批次、报告和删除批次写执行摘要；SHA、媒体、预览、MySQL、RocksDB、报告与删除失败及时刷盘。
7. SHA-512 后的媒体能力解析已改为本地无效时继续查询 MySQL；远端有效结果回填 RocksDB。图片零值、视频任一帧零值、算法版本不匹配均不复用。
8. 新增报告摘要/按序号读取、报告媒体字段、执行日志 54 条及时刷盘、视频零帧无效和配置往返测试；Debug 与 Release 均为 33/33 通过。
