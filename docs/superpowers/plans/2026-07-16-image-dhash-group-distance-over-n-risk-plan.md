# 图片相似组内距离超过 n 的其他风险排查与修复计划

> 日期：2026-07-16  
> 状态：已完成只读排查，等待执行  
> 目标：排除动态分桶和完整链接算法之外，仍可能让界面显示“同组某些组合汉明距离大于当前 n”的因素。

## 1. 结论

有其他因素，其中当前项目最需要优先处理的是“报告规则元数据缺失和旧报告继续复用”。

当前新生成流程已经对候选组全部视觉签名成员执行完整链接校验，但持久化报告只记录：

```text
algorithm_version = media-dhash-v2
```

没有记录：

- 生成报告时的图片最大汉明距离 n。
- 图片分桶格式版本。
- 图片分组规则版本。
- 是否采用完整链接分组。
- 报告生成程序版本。

GUI 加载报告时只读取 active generation 和组数据，不会核对当前配置 n，也不会重新验证组内任意两项。因此旧规则报告、不同 n 的报告或外部损坏报告都可能继续显示。

## 2. 高优先级因素

### 2.1 旧并查集报告仍被加载

旧实现按相似边连通分量合组，可能存在：

```text
A≈B
B≈C
A≉C
```

旧报告和新报告使用相同的报告组编码版本及 `media-dhash-v2` 算法文本。升级程序后，旧报告不会自动失效。

表现：

- 重新打开程序后仍看到修改前的大组。
- 用户提供的多个 dHash 两两距离超过 n，但仍属于同组。
- 删除旧报告并重新生成后现象消失。

风险等级：高。

### 2.2 修改 n 后继续查看旧报告

报告生成线程会冻结启动时的 n，这一点是正确的；但报告发布后没有保存该 n。

例如：

```text
报告按 n=8 生成
用户随后把配置改为 n=4
GUI 继续加载 n=8 的报告
```

此时组内距离 5～8 在原报告中合法，但相对当前配置 n=4 已超限。

风险等级：高。

### 2.3 报告没有完整链接规则版本

`media-dhash-v2` 主要表示媒体特征算法版本，没有区分：

- 旧版连通分量分组。
- 代表中心分组。
- 严格完整链接分组。

只要组二进制编码仍可解析，GUI 就会接受报告，无法判断其分组语义是否过期。

风险等级：高。

## 3. 中优先级因素

### 3.1 GUI 阈值文字固定为 `< 5`

重复组详情当前固定显示：

```text
阈值严格 < 5
```

该文本只在默认 n=4 时与 `distance <= n` 等价。n 改成其他值后，界面仍显示 `< 5`，会导致用户使用错误阈值核对结果。

风险等级：中。它通常不改变分组数据，但会造成“结果超过 n”或“结果不符合界面阈值”的认知错误。

### 3.2 报告加载时不做完整性复核

新报告生成阶段会逐组执行完整链接校验，但加载已有报告时只反序列化成员和证据，不重新验证：

```text
对组内任意两个不同图片成员：
distance <= report.n
```

以下情况可能绕过生成阶段保证：

- RocksDB 报告数据损坏。
- 旧版本程序写入的报告。
- 手工迁移或复制了不兼容的报告目录。
- 将来修改报告编码但没有同步提升规则版本。

风险等级：中。

### 3.3 证据只保存部分边，不保存全部两两结果

生成阶段实际对候选组全部成员做校验，但报告只保存用于解释分组的部分证据边，不保存 O(m²) 的全部两两证据。

因此：

- 详情窗口中的证据数量不能证明已经展示全部比较。
- 若报告来自旧版本，单看证据无法确认组内所有组合都合格。
- 当前缺少一个轻量的组级完整性摘要，例如“最大两两距离”和“校验时的 n”。

风险等级：中。

### 3.4 运行了其他输出目录中的旧程序

项目存在 Debug、Release 等不同输出目录。如果实际启动的是未更新的旧 `VideoScGUI.exe`，它仍可能用旧并查集规则生成报告。

风险等级：中。

建议在界面显示：

- 程序构建版本。
- 报告规则版本。
- 报告生成时的 n。
- 报告 generation ID 和生成时间。

## 4. 数据层因素

### 4.1 文件在报告生成后发生变化

报告成员保存的是生成时 MySQL 内容记录中的 SHA-512 和 dHash。报告生成后，磁盘路径指向的文件可能被覆盖或修改。

如果用户重新读取当前文件计算 dHash，而不是使用报告内保存的 dHash，则可能出现：

```text
报告内旧 dHash 两两 <= n
当前磁盘文件的新 dHash 距离 > n
```

风险等级：中。

详情窗口应明确区分：

- 报告快照 dHash。
- 当前文件重新计算 dHash。
- 文件是否已改变。

必要时用文件大小、最后修改时间和 SHA-512 做重新验证。

### 4.2 MySQL 视觉特征数据错误或算法版本标签复用

报告只读取：

```text
media_algorithm_version = media-dhash-v2
```

如果数据库中的 dHash 被错误写入、人工修改，或在 dHash 生成语义变化后仍复用同一算法版本文本，报告会依据错误特征分组。

不过，如果界面显示的正是报告生成时使用的同一个错误 dHash，完整链接仍应保证这些显示值两两不超过 n。该因素更容易表现为“文件实际画面不相似”，而不是“显示的 dHash 距离超过 n”。

风险等级：低至中。

## 5. 当前实现中不会单独导致超限的因素

### 5.1 分桶碰撞

同桶只代表候选，不代表最终相似。当前流程仍执行完整 64 位汉明距离复核，因此桶碰撞只会增加计算量，不应让距离大于 n 的成员加入组。

### 5.2 相同视觉签名折叠

图片视觉签名包含完整 64 位 `image_dhash`。多个 SHA-512 内容被折叠到同一视觉代表时，它们的 dHash 完全相同，彼此距离为 0。

当前定义下，仅校验该完整签名代表是安全的，不等于用基准 dHash 对不同签名进行分组。

### 5.3 同 SHA-512 的多条路径

相同 SHA-512 表示相同文件内容。路径展开会增加组内文件数量，但不会增加不同 dHash 组合。

如果同 SHA-512 在报告中出现不同 dHash，则属于数据库或报告数据损坏，不是正常路径展开。

### 5.4 贪心归组顺序

稳定贪心完整链接可能影响“一个合法内容最终进入哪个组”，也不保证组数最少，但只要每次加入前都校验全部组成员，就不会破坏组内任意两项 `<= n` 的不变量。

## 6. 修复方案

### 6.1 报告保存规则元数据

为 dHash 相似报告增加元数据：

```text
report_codec_version
similarity_grouping_version
image_bucket_index_version
image_max_hamming_distance
media_algorithm_version
generator_build_version
generated_utc_ms
```

建议分组规则版本使用独立稳定值，例如：

```text
image-complete-link-v1
```

不能继续只依赖 `media-dhash-v2`。

### 6.2 加载时执行兼容校验

加载 dHash 报告前核对：

1. 报告是否包含 n。
2. 报告分组规则是否为当前支持的完整链接版本。
3. 报告桶格式是否受当前程序支持。
4. 报告媒体算法版本是否匹配。

旧报告处理方式：

```text
标记为规则过期
禁止作为当前配置结果展示
提示用户重新生成
```

不建议把旧报告静默当作新规则报告。

### 6.3 配置变化时标记报告过期

当用户保存不同的 n 后：

- 不立即删除旧报告。
- 在界面将旧报告标记为“按 n=旧值生成”。
- 默认不把旧报告解释为当前 n 的结果。
- 提供“重新生成 dHash 报告”操作。

如果产品希望只保留当前配置结果，也可在 n 变化后撤销 active similar report，但删除报告属于状态变更，应在实施前再次确认。

### 6.4 保存组级完整性摘要

图片组建议额外保存：

```text
distinct_content_count
maximum_pairwise_hamming_distance
validated_pair_count
validation_threshold_n
```

发布报告前必须满足：

```text
maximum_pairwise_hamming_distance <= validation_threshold_n
```

详情窗口显示：

```text
生成阈值 n
组内最大两两距离
已校验不同内容对数量
```

不需要保存全部 O(m²) 证据，只保存统计摘要即可。

### 6.5 发布前增加最终完整性审计

在组写入正式报告前，对每个图片组执行一次独立审计：

1. 按不同 SHA-512 去重。
2. 对所有不同 dHash 组合计算汉明距离。
3. 发现任意距离超过 n 时拒绝发布整份报告。
4. 写入失败日志，包含组 ID、两个 SHA-512、两个 dHash、实际距离和 n。

该审计会增加计算量，但它是最可靠的最终防线。可复用生成阶段已经驻留的紧凑 dHash 列表，避免重新读取文件或 MySQL。

### 6.6 修正 GUI 口径

把固定文字：

```text
阈值严格 < 5
```

改为报告元数据驱动：

```text
生成阈值：汉明距离 <= n
```

若报告没有 n：

```text
旧报告未记录阈值，请重新生成
```

## 7. 测试计划

1. 用旧并查集报告 fixture 加载，必须提示规则过期而不是正常展示。
2. 用 n=8 生成报告，再把当前配置改为 n=4；界面必须明确显示报告按 n=8 生成。
3. 当前配置与报告 n 一致时正常加载。
4. 报告缺少 grouping version 时拒绝作为当前报告。
5. 人工构造组内一对距离大于 n 的报告，加载或发布审计必须拒绝。
6. GUI 阈值文本随报告 n 变化，不再固定显示 `<5`。
7. 同一 dHash 的多个 SHA-512 内容仍能通过最终审计。
8. 同 SHA-512 多路径不会重复计入 `validated_pair_count`。
9. 最终审计支持取消并及时退出。
10. Debug/Release x64 构建成功，相关测试全部通过。

## 8. 建议排查顺序

针对当前已经看到的异常组，先执行只读确认：

1. 确认正在运行的 `VideoScGUI.exe` 绝对路径和构建时间。
2. 确认该组来自重新生成后的报告，而不是升级前 active report。
3. 记录当前配置 n。
4. 记录报告 generation ID。
5. 导出该组所有不同 SHA-512 与报告内 dHash。
6. 对组内所有不同 dHash 两两计算距离，输出最大距离和超限组合。
7. 删除旧 dHash 报告后重新生成，再对同一组复核。

如果重新生成后的新报告仍存在超限组合，优先检查：

- 运行程序是否确实是新构建。
- 最终分组阶段是否被其他代码路径绕过。
- RocksDB active generation 是否发布了预期的新 generation。
- 报告写入后是否被旧进程覆盖。

当前仅生成排查与修复计划，不执行代码修改。
