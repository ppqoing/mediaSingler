# dHash 相似报告超大连通组修复计划

> 日期：2026-07-16  
> 状态：已执行完成
>
> 决策：同一相似组内任意两个不同 SHA-512 内容都必须直接通过当前 dHash 相似判定；不再采用代表中心分组。本文第 4 节中的方案比较仅作为决策记录，第 11 节为最终执行口径，并覆盖前文“推荐代表中心”的实施描述。

## 1. 问题现象

dHash 相似报告中存在包含数百个文件的重复组。用户从同一组截取的四个图片 dHash 为：

```text
0x68e0e0e0f0f0f0f0
0xe0e0f2b0a0e0e0e0
0xe4e0e0f0f0f0f8e8
0xf8fcd4f8f0f0f8f8
```

按项目当前 64 位 dHash 汉明距离算法计算，两两距离如下：

| dHash 对 | 汉明距离 |
| --- | ---: |
| 1 ↔ 2 | 11 |
| 1 ↔ 3 | 7 |
| 1 ↔ 4 | 12 |
| 2 ↔ 3 | 10 |
| 2 ↔ 4 | 17 |
| 3 ↔ 4 | 11 |

当前图片相似阈值为：

```text
汉明距离 <= 4
```

因此这四个 dHash 中没有任何一对满足直接相似条件，却仍出现在同一报告组。

## 2. 根因结论

### 2.1 当前报告把“相似关系”按连通分量分组

`DuplicateReportGenerator::GenerateSimilar()` 对每一条通过真实距离校验的相似边调用：

```text
UnionDigests(left, right)
```

`UnionDigests()` 使用 RocksDB 并查集合并摘要。最终报告按并查集根节点输出组，因此当前分组语义实际是：

```text
只要两个内容之间存在一条或多条相似边组成的路径，就属于同一组。
```

例如：

```text
A 与 B 距离 4
B 与 C 距离 4
C 与 D 距离 4
```

即使：

```text
A 与 C 距离 7
A 与 D 距离 12
```

并查集仍会把 `A、B、C、D` 合并为一个组。这属于单链接聚类的链式效应。

用户提供的四个值两两均不相似，说明它们之间存在其他未截取的中间内容，把原本差异很大的两端通过多条合法相似边连接到了同一连通分量。

### 2.2 候选索引和真实距离复核本身没有放宽阈值

当前图片最终比较仍调用：

```text
DHashRules::ImagesAreSimilar(left, right)
```

阈值仍为距离不超过 `4`。MIH 候选索引只负责召回候选，最终边在写入前会进行真实距离复核。

因此问题不在“单条错误相似边”，而在“正确的局部边被并查集无限传递合并”。

### 2.3 最终按路径展开会进一步放大成员数量

相似计算首先按 SHA-512 内容建立节点，但第四阶段会读取 MySQL 全部 active 路径，并把属于同一内容根节点的每条路径都写入组成员：

```text
一个 SHA-512 内容节点
    ├─ 路径 1
    ├─ 路径 2
    └─ 路径 3
```

所以超大组数量由两部分叠加：

1. dHash 内容节点因链式传递形成大型连通分量。
2. 每个内容节点可能对应多个 SHA-512 精确重复路径。

因此界面中的“数百个文件”不一定等于数百个不同 dHash，但用户给出的四个差异较大的 dHash 已足以证明组内存在链式扩张。

## 3. 修复目标

1. 同一图片相似组不能仅凭间接相似路径无限扩张。
2. 组内任意两个不同 SHA-512 内容必须满足直接 dHash 相似条件。
3. 用户查看同组任意两个成员时，不再出现完全无法解释的巨大差异。
4. 保留当前汉明距离阈值 `<= 4`，本次不通过降低阈值掩盖分组问题。
5. 保留 MIH 候选召回和最终真实距离复核。
6. 相同 SHA-512 的多个文件路径仍可显示在同一内容分组下。
7. 不对全库内容进行无条件全量两两比较；只对索引召回的候选及候选组成员执行必要的完整链接校验。

## 4. 分组方案比较

### 4.1 方案 A：代表中心分组

为每个组选择一个稳定代表 dHash。新内容只有与代表直接相似时才能加入该组：

```text
distance(member, representative) <= 4
```

如果一个内容与现有代表均不相似，则新建另一个组。

优点：

- 能彻底阻断长链无限扩张。
- 每个成员都能用一条直接证据解释为什么进入该组。
- 不需要对组内所有成员两两比较。
- 可继续使用当前磁盘候选索引。
- 适合数十万或更多内容的全量报告。

限制：

- 同一组中两个非代表成员之间的距离可能大于 `4`，最大可能达到 `8`。
- 内容处理顺序可能影响分组结果，因此必须定义稳定代表与稳定排序。

推荐使用稳定规则：

```text
代表 = 组内 SHA-512 字典序最小的内容
内容处理顺序 = SHA-512 字典序
```

这是本计划的推荐方案。

### 4.2 方案 B：完全链接分组

新内容加入组前，必须与组内每一个已有内容直接相似：

```text
对所有 member：
distance(new, member) <= 4
```

优点：

- 保证同组任意两项都满足当前阈值。
- 语义最严格、最容易理解。

缺点：

- 大组可能退化为大量两两比较。
- 必须额外持久化或反复加载组内所有 dHash。
- 报告生成时间可能明显增加。
- 视频需要逐项执行六帧和时长比较，成本更高。

### 4.3 方案 C：仅限制最大组大小

当组达到指定数量后强制拆组。

该方案不推荐，因为：

- 不能解决同组内容实际不相似的问题。
- 拆分边界没有视觉语义。
- 不同扫描顺序可能得到完全不同的结果。
- 只是隐藏超大组现象。

## 5. 推荐设计：稳定代表中心分组

### 5.1 图片内容分组规则

1. MIH 索引继续召回所有距离可能不超过 `4` 的候选。
2. 对候选执行真实汉明距离复核。
3. 不再立即对每条合法边执行全局并查集合并。
4. 每个内容查找其可以直接匹配的稳定代表。
5. 如果存在多个可匹配代表，选择：
   - 汉明距离最小者。
   - 距离相同时选择 SHA-512 字典序最小者。
6. 没有可匹配代表时，把当前内容登记为新代表。
7. 报告组只包含代表与其直接成员。

组内保证：

```text
每个内容与代表的 dHash 距离 <= 4
```

### 5.2 视频内容分组规则

视频同样改为代表中心分组，但加入组时仍必须满足：

- 两个视频时长差不超过当前限制。
- 六帧 dHash 均有效。
- 六帧平均汉明距离满足当前阈值。
- 静态视频不参与相似报告。

每个视频成员必须与代表视频直接通过 `CompareVideos()`。

图片与视频不能进入同一组。

### 5.3 相同视觉签名处理

当前相同 dHash 会先压缩为代表内容。该优化可以保留：

- dHash 完全相同的多个 SHA-512 内容先映射到同一个视觉签名代表。
- 路径展开时仍保留全部文件路径。
- 视觉签名代表再参与中心分组。

需要避免把“相同签名压缩代表”和“相似组中心代表”混为同一种 RocksDB 键。

### 5.4 报告证据

每个非代表内容至少保存一条：

```text
代表内容 ↔ 当前内容
```

的真实距离证据。

详情窗口新增或明确显示：

- 组代表文件。
- 当前成员到代表的汉明距离。
- 视频成员到代表的六帧距离和平均距离。

删除策略不应默认把代表标记为必须永久保留；代表仅用于解释分组，不等于用户最终选择保留的文件。

### 5.5 路径数量与内容数量分别展示

报告摘要和详情建议同时展示：

```text
内容数：按不同 SHA-512 统计
文件数：按 active 路径统计
```

这样可以区分：

- 一个视觉内容存在大量精确副本。
- 大量不同内容被相似规则聚到一起。

排序中的“组内文件数量”继续按路径数；可新增按不同内容数排序，但不在本次默认范围内。

## 6. 数据结构调整

建议在相似报告工作命名空间增加：

```text
representative/<media-kind>/<sha512> -> representative-sha512
representative-member/<representative-sha512>/<sha512> -> evidence
```

最终 `DuplicateGroup` 建议增加：

```text
representative_sha512
distinct_content_count
```

如果不希望升级持久化报告编码，可在报告生成阶段将代表对应的首个 `path_id`
写入现有 `retained_path_id` 以外的新字段；不能复用 `retained_path_id`，因为它属于删除计划。

报告格式发生变化时需要：

- 提升组编码版本。
- 旧报告继续按旧版本只读。
- 旧报告详情不显示代表信息。
- 新生成报告使用新分组语义。

2. 确定采用“代表中心”还是“完全链接”分组语义。

8. 详情窗口展示代表及成员到代表距离。

   - 同组每个图片成员与代表距离均不超过 `4`。
   - 结果不受输入顺序影响。
   - 相同 dHash 的不同 SHA-512 内容保持同组。
   - 同 SHA-512 的多条路径全部保留。

    - 报告生成耗时。

## 9. 验收标准

1. 用户提供的四个 dHash 不会仅因间接链路全部进入同一组。
2. 同组任意两个不同图片内容的汉明距离均不超过 `4`。
3. 同组任意两个不同视频内容均通过 `CompareVideos()`。
4. 相似报告结果在相同数据下可重复生成，且不受 MySQL 返回顺序影响。
5. 详情界面能够显示选中内容的 dHash，并支持核对任意两项的实际距离。
6. 文件数和不同 SHA-512 内容数分别显示。
7. 同一内容的大量路径不会被误解为大量不同视觉内容。
8. 不执行全库无条件两两比较，只对候选组执行完整链接校验。
9. Debug/Release x64 构建成功，全部测试通过。

## 10. 已确认决策

采用最严格方案：同组任意两个不同内容之间都必须直接满足相似条件，并接受候选组内完整链接校验带来的额外报告生成成本。

## 11. 已确认的最终执行方案

用户已确认采用方案 B：完整链接分组。执行时以下规则覆盖第 5、6、8、9 节中以“代表中心”为前提的描述。

### 11.1 分组不变量

对于同一图片相似组中的任意两个不同 SHA-512 内容 `a`、`b`，必须满足：

```text
HammingDistance(a.dhash, b.dhash) <= 4
```

对于同一视频相似组中的任意两个不同 SHA-512 内容 `a`、`b`，必须满足：

```text
CompareVideos(a, b) == similar
```

其中仍沿用现有视频时长、六帧有效性、六帧距离及静态视频排除规则。图片与视频不得进入同一组。

文件路径不是完整链接校验节点。相同 SHA-512 对应的多条 active 路径在内容分组完成后统一展开，因此不会重复参与 dHash 比较。

### 11.2 确定性完整链接归组算法

1. 按媒体类型隔离处理图片和视频。
2. 对不同 SHA-512 内容按 SHA-512 字典序稳定排序，消除 MySQL 返回顺序和线程完成顺序对结果的影响。
3. 继续使用 MIH 等现有索引召回与当前内容直接相似的候选内容，不进行全库无条件两两比较。
4. 根据候选内容反查可能接纳当前内容的候选组；没有任何直接相似候选的组不进入校验。
5. 当前内容加入某一候选组前，必须与该组全部不同 SHA-512 成员逐一执行真实相似判定。
6. 只有全部比较通过，当前内容才能加入该组；任意一项不通过就拒绝加入该组。
7. 如果多个组都可以接纳当前内容，按以下固定优先级选择：
   - 当前内容到组内成员的最大距离最小；
   - 最大距离相同时，平均距离最小；
   - 仍相同时，选择组内最小 SHA-512 字典序更小的组。
8. 没有任何组能够完整接纳时，新建一个组。
9. 已完成的分组不再通过组间并查集合并，防止重新引入传递闭包。

该算法属于确定性的贪心完整链接划分。它保证最终每个组满足“任意两项直接相似”，但不保证在所有可能划分中组数绝对最少；后者属于计算代价很高的全局组合优化，不纳入本次范围。

### 11.3 性能控制

- 图片优先使用 64 位 dHash 的按位异或和 popcount 校验，完整链接比较只在候选组内部进行。
- 为每个构建中的组保留紧凑 dHash 成员集合，避免为每次校验重复读取 RocksDB 或 MySQL。
- 视频先执行时长和无效帧快速拒绝，再执行六帧距离计算。
- 视频组缓存六帧 dHash 与时长，不在组内校验时重复读取缩略图或媒体文件。
- 保留现有取消信号检查；组内成员循环也必须响应取消，避免大候选组导致取消延迟。
- 进度统计增加“候选组完整链接校验”工作量，避免界面长时间停留在同一百分比。
- 记录候选边数、完整链接比较次数、拒绝加入次数、最大不同内容数和最大路径数，便于判断性能及分组质量。

### 11.4 报告与界面口径

- 报告继续展示同一 SHA-512 内容对应的全部 active 文件路径。
- 组摘要区分：
  - `不同内容数`：不同 SHA-512 数量。
  - `文件数`：展开后的 active 路径数量。
- 即使严格分组后仍可能出现数百个文件的组：例如大量完全相同内容路径，或者组内所有不同内容确实两两满足阈值。这种情况属于合法结果，不再按数量强制拆组。
- 详情窗口无需显示“代表文件”，但应能够显示选中任意两项的 dHash 及实际距离。

### 11.5 最终测试与验收

必须增加并通过以下测试：

1. `A≈B`、`B≈C`、`A≉C` 时，三者不得全部进入同一组。
2. 遍历每个图片结果组的任意两项，汉明距离都不得超过 `4`。
3. 遍历每个视频结果组的任意两项，都必须通过 `CompareVideos()`。
4. 用户提供的四个 dHash 两两距离均超过 `4`，不得出现在同一图片相似组。
5. 相同输入以不同顺序执行，分组结果必须一致。
6. 相同 SHA-512 的多条路径全部保留，但只参与一次内容级相似计算。
7. 组内校验过程中取消报告生成，任务应及时停止且不发布半成品报告。
8. 旧版报告仍可只读打开；新报告重新生成后使用严格分组语义。
9. 对当前 MySQL 全量数据分别记录修改前后的最大文件数、最大不同内容数、组数、生成耗时和完整链接比较次数。

本节方案已按后续动态分桶要求一并执行，最终结果见第 13 节。

## 12. 新增需求：图片 dHash 动态分桶与剪枝

### 12.1 需求范围

本节只调整图片 dHash 候选召回和图片真实汉明距离校验。视频仍沿用六帧、时长桶及 `CompareVideos()` 规则，除非用户后续明确要求视频也使用相同的动态分段参数。

图片使用配置项 `n` 控制相似阈值与分桶数量：

```text
分段数量 = n + 1
```

把 64 位 dHash 按从高位到低位的固定顺序切成 `n + 1` 个连续、不重叠且非空的段。段宽尽量均匀：

```text
基础段宽 = 64 / (n + 1)
余数 = 64 % (n + 1)
前“余数”段比其余段多 1 位
```

例如 `n = 4` 时：

```text
64 位 -> 13、13、13、13、12 位
```

每个图片 dHash 按段序号分别写入对应桶：

```text
bucket/<index-version>/<n>/<segment-index>/<segment-width>/<segment-value>
```

查询时同样依次计算 `n + 1` 个段，只扫描“段序号、段宽、段值”完全一致的桶。只要候选与查询值至少命中一个相同段，才进入后续真实距离计算。

### 12.2 正确性依据与已确认边界

鸽巢原理保证：

```text
若 64 位 dHash 的汉明距离 <= n，
则切成 n + 1 个不重叠段后，至少有一个段完全相等。
```

因此在采用“距离小于等于 n”时，该分桶不会漏掉合法候选。

用户已确认配置语义为：

```text
distance <= n
```

该语义与项目当前 `<= 4` 一致。默认 `n = 4`，切 `n + 1 = 5` 段，配置页面显示值就是允许的最大汉明距离，不存在相差 1 的换算。

用户已确认配置范围为 `0..15`，默认值为 `4`。配置校验和 GUI 数值控件必须使用相同边界。超过范围的配置禁止保存或启动任务，不能静默截断。

### 12.3 替换当前固定索引

当前实现并非直接的 `n + 1` 单段桶：

- 图片固定写入 6 段中任意两段的 15 个联合索引。
- 查询固定扫描 15 个两段前缀。
- `DHashRules::kMaximumHammingDistance` 固定为 `4`。
- `Split()` 固定返回 5 段，`SplitPairs()` 固定返回 15 个段对。

执行时改为：

1. 新增只包含图片相似配置的 `DHashSimilarityConfig`，至少保存 `image_max_hamming_distance`。
2. 动态生成 `n + 1` 个 `DHashSegment`，不再为图片生成两段组合。
3. 每个唯一图片签名只写 `n + 1` 个单段索引条目。
4. 查询最多扫描 `n + 1` 个单段桶，并对重复命中的候选 SHA-512 去重。
5. 候选仍必须执行真实汉明距离校验；分桶命中不能直接判为相似。
6. 视频现有 `SplitPairs()` 和联合索引暂时保留，避免扩大本次修改范围。

### 12.4 剪枝顺序

图片候选和完整链接归组按以下顺序剪枝：

1. **媒体类型剪枝**：仅图片进入图片桶。
2. **无效值剪枝**：缺失或默认 dHash 不写桶、不参与报告。
3. **桶命中剪枝**：没有任何相同段的内容不读取完整内容记录。
4. **候选去重**：同一候选命中多个桶时只计算一次。
5. **自身与相同 SHA-512 剪枝**：内容自身不比较；同 SHA-512 多路径只展开、不重复计算。
6. **完整值快速判等**：两个 dHash 完全相等时距离直接为 0。
7. **分段累计距离早停**：真实校验按段累计 popcount，累计距离一旦超过 n 立即停止，不再处理剩余段。
8. **完整链接组剪枝**：新成员加入候选组时，优先检查历史上更容易拒绝候选的成员；任意一项距离超过 n，立即拒绝整个候选组。
9. **取消剪枝**：扫描桶、遍历候选和遍历组成员时均检查取消标志。

对于 64 位数据，单次 `__popcnt64(left ^ right)` 本身很快。实现时需基准比较“单次完整 popcount”和“逐段早停”，只保留实际更快的路径；无论采用哪种实现，最终判定结果必须完全一致。

### 12.5 配置、快照与索引兼容

计划新增 JSON 配置分区：

```json
{
  "dhash_similarity": {
    "image_max_hamming_distance": 4
  }
}
```

配置页面增加“图片 dHash 最大汉明距离 n”，旁边说明：

```text
索引会分为 n+1 段；n 越大，召回候选越多、报告生成越慢。
```

配置修改后的生效规则：

- 正在运行的扫描或报告继续使用启动时快照。
- 下一次扫描和下一次 dHash 报告使用新值。
- 扫描断点保存 n，恢复时继续使用原值，不能静默改用当前页面的新值。
- 报告生成线程必须捕获配置快照，不能直接读取可变 GUI 配置。

分桶索引键必须包含：

```text
图片索引格式版本 + n + 分段布局版本
```

禁止不同 n 共享同一 RocksDB 前缀。配置变化后：

- 报告生成的临时候选索引使用新的命名空间重建，成功发布或取消后按现有生命周期清理。
- 持久扫描索引不能仅切换前缀后继续增量写入，否则历史内容没有新 n 对应的桶，会产生漏召回。
- 若持久扫描索引仍被业务使用，则需要增加“按当前 n 重建全部图片桶”的显式阶段或停止依赖该持久索引；不能读取不完整的新前缀。

配置模式需从版本 1 升级到版本 2，并提供 1 到 2 的迁移：

```text
旧配置缺少该字段 -> image_max_hamming_distance = 4
```

不能继续沿用当前“旧版本配置直接拒绝”的行为，否则升级程序后已有配置会失效。

### 12.6 与最严格完整链接分组的关系

动态分桶只负责召回“可能距离不超过 n”的候选，不能代替第 11 节的完整链接约束。

最终图片组仍必须满足：

```text
对组内任意不同内容 a、b：
HammingDistance(a.dhash, b.dhash) <= n
```

新内容即使命中了组内某个成员的桶，也必须与该组其余全部不同 SHA-512 内容逐一通过真实距离校验，才能加入该组。
