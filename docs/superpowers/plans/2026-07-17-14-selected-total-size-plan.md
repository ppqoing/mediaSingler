# 14－显示选中文件数量与总体积修改计划

> 日期：2026-07-17  
> 状态：已执行，等待最终验收
> 数据来源：持久化报告选择 metadata

## 1. 修改目标

重复报告页面和永久删除确认窗口实时显示当前报告中选中文件的数量、总体积和涉及组数。

显示示例：

```text
已选择 1,234 个文件，共 87.6 GiB，涉及 318 个重复组
```

## 2. 当前问题

- 选择状态主要保存在 `m_reportSelections` 和可见组缓存。
- 没有全报告选择字节汇总。
- 每帧遍历全部报告成员会造成明显性能问题。
- 删除确认窗口无法提前显示准确总体积。

## 3. 持久化汇总

`ReportSelectionMetadata` 保存：

```text
selected_file_count
selected_total_bytes
selected_group_count
source_report_generation
updated_utc_ms
```

每个组摘要保存：

```text
group_id
selected_file_count
selected_total_bytes
```

## 4. 更新规则

### 单文件切换

使用同一个 RocksDB `WriteBatch`：

1. 写入或删除成员选择键。
2. 更新组文件数和字节数。
3. 更新全局文件数、总体积和组数。
4. 提交成功后发布新快照。

### 按组或全报告选择

- 后台流式计算每个组选择。
- 在 selection staging generation 中累计汇总。
- 全部完成后原子发布 active selection 和最终 metadata。
- GUI 不显示半成品汇总。

### 删除后

- 实际删除成功的成员从选择中移除。
- 跳过和失败成员保留选择，便于用户再次排查。
- 使用原子批处理修正摘要。

## 5. GUI 修改

在重复报告操作区显示：

- 已选择文件数。
- 已选择总体积。
- 涉及重复组数。

在永久删除确认窗口显示同一快照，并在删除过程中同时显示：

- 原始选择总体积。
- 已处理体积。
- 已成功释放体积。

所有体积使用现有 `FormatBytes()`，不自动换行。

## 6. 摘要修复

如果 metadata 缺失或与成员键不一致：

1. 禁止直接开始危险删除。
2. 后台流式扫描选择成员键。
3. 重建组摘要和全局摘要。
4. 写入文本日志，记录修复前后数量和字节。

## 8. 性能约束

- GUI 每帧只读取一个汇总快照。
- 不因滚动、排序或切换详情重新遍历全部组。
- 大规模全选在后台流式执行并显示选择进度。
- 计数使用 64 位无符号整数，并检测字节累加溢出。

## 9. 测试与验收

1. 单个选择和取消后数量、字节、组数准确。
2. 当前可见组批量选择汇总准确。
3. 全报告选择与逐成员求和完全一致。
4. 重启后汇总与选择状态一致。
5. 成功删除后汇总减少；失败成员仍保留。
6. 摘要损坏时阻止删除并可重建。
7. 数百组、数万成员时 GUI 不进行每帧全量遍历。

## 10. 不在本条范围内

- 不跨 SHA-512 和 dHash 两个报告合并统计。
- 不统计跨严格组只读关系成员。
- 不把理论可释放大小当作实际选中总体积。
