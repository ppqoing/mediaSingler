# 图片 dHash 相似文件对漏筛风险排查与修改计划

> 日期：2026-07-16  
> 状态：方案已确认，采用“非重叠严格主分组 + 完整直接相似关系”，详细实施步骤见同目录实施计划  
> 判定口径：沿用已确认规则，图片相似条件为 `HammingDistance(left, right) <= n`

## 1. 结论

会出现“两个 dHash 的汉明距离 `<= n`，但最终没有在同一个重复组中显示”的情况。

需要区分两个层面：

1. **候选召回层**：在两个 dHash 都有效、使用同一个 `n`、相同分段规则并且都已写入本次索引的前提下，当前 `n+1` 分桶算法不会漏掉距离 `<= n` 的候选对。
2. **最终分组层**：当前实现要求每个内容只能归入一个组，同时要求组内任意两个内容的距离都 `<= n`。这种“非重叠严格组”会不可避免地丢失一部分直接相似关系。

典型示例：

```text
distance(A, B) <= n
distance(B, C) <= n
distance(A, C) > n
```

如果按 `A、B、C` 的顺序处理：

```text
A 创建组 G1
B 加入 G1
C 与 B 相似，但 C 与 A 不相似，因此 C 不能加入 G1
C 创建组 G2
```

最终 `B` 和 `C` 虽然满足 `distance <= n`，却不在同一结果组。

这不是分桶漏召回，而是以下三个要求无法同时满足：

1. 每个文件只能出现在一个组中。
2. 每个组内任意两个文件都满足 `distance <= n`。
3. 每一对满足 `distance <= n` 的文件都至少在一个共同组中出现。

对于上面的 `A-B-C` 关系，如果必须完整保留两条相似关系，只能允许 `B` 同时出现在 `{A, B}` 和 `{B, C}` 两个重叠组中，或者另外输出相似文件对。

## 2. 当前候选分桶为什么正常情况下不会漏召回

当前图片 dHash 为 64 位，按 `n+1` 个连续且不重叠的分段建立候选桶：

```text
segment_count = n + 1
```

如果两个 dHash 的汉明距离 `<= n`，最多只有 `n` 个分段包含差异位，因此至少有一个同序号分段完全相同。

当前实现已经具备以下条件：

- `DHashRules::Split` 覆盖全部 64 位，没有重叠或遗漏。
- 写索引和查询候选都调用同一个 `Split`。
- 桶前缀包含 `n`、分段序号、分段宽度和分段值。
- 查询遍历全部 `n+1` 个桶。
- 同一个候选命中多个桶时进行去重。
- 命中桶后仍执行完整 64 位汉明距离复核。

对应代码位置：

- `DedupCore/dedup/DHashSimilarity.cpp:98`
- `DedupCore/dedup/DHashSimilarity.cpp:204`
- `DedupCore/dedup/DHashSimilarity.cpp:240`

因此，若只讨论“两个有效 dHash 是否能互相进入候选比较”，数学上和当前分段实现上都能保证无漏召回。

## 3. 当前严格分组造成相似对分离的原因

当前报告生成按稳定 SHA-512 顺序逐个处理视觉代表：

1. 只从已经处理过的候选中查找其所属组。
2. 对候选组内的全部成员执行完整链接校验。
3. 新内容只有与组内每一个成员都相似时才能加入。
4. 多个组都可加入时只选择一个最优组。
5. 没有组可加入时创建新组。
6. 每个内容只保存一个 `group-of` 归属。

对应代码位置：

- 候选仅引用已完成归组的历史代表：`DedupCore/dedup/DuplicateReportService.cpp:1112`
- 读取候选当前所属的唯一组：`DedupCore/dedup/DuplicateReportService.cpp:1135`
- 对候选组全部成员做完整链接校验：`DedupCore/dedup/DuplicateReportService.cpp:1168`
- 多个合法组只选择一个最优组：`DedupCore/dedup/DuplicateReportService.cpp:1225`
- 写入唯一 `group-of`：`DedupCore/dedup/DuplicateReportService.cpp:1241`

这种实现能保证：

```text
同组任意两个成员的距离都 <= n
```

但不能保证：

```text
任意距离 <= n 的两个成员最终都在同一组
```

处理顺序只会改变哪些直接相似关系被分开，不会消除这种结构性冲突。

## 4. 会造成真正候选漏筛或结果不可见的其他因素

### 4.1 dHash 无效或缺失

以下内容不会写入图片候选索引：

- 媒体类型不是图片。
- `image_dhash` 为空。
- `image_dhash == 0`。

对应代码：`DedupCore/dedup/DHashSimilarity.cpp:204`。

因此即使重新读取当前文件后算出的两个 dHash 距离很小，只要数据库中的其中一条记录为空或为默认值，本次报告也不会比较它们。

### 4.2 媒体算法版本不匹配

报告只读取指定 `media_algorithm_version` 的记录，并要求图片 dHash 非空且非零。

对应代码：

- `DedupCore/persistence/MySqlReadRepository.cpp:346`
- `VideoScGUI/VideoScApp.cpp:1422`

一条记录仍使用旧算法版本、错误版本或失败版本时，不会进入本次相似报告。

### 4.3 数据尚未同步到 MySQL

当前 dHash 总结果报告以 MySQL 数据为来源。以下数据不会出现在本次报告：

- 仍只保存在本地嵌入式数据库中的新计算结果。
- 尚未达到普通批量同步门槛且任务仍在运行的数据。
- 同步失败、等待重试的数据。
- 扫描或计算尚未结束的数据。

如果在扫描、计算或同步完成前生成报告，会出现数据库中缺少其中一方的情况。

### 4.4 没有可用的活动文件路径

报告先按内容 SHA-512 和 dHash 分组，再通过活动路径展开为界面成员。

以下情况可能导致内容层已经匹配，但界面没有显示成有效重复组：

- 其中一方的路径已被删除。
- 路径状态没有同步为活动状态。
- 两个内容展开后不足两个有效文件路径。

对应代码：

- `DedupCore/dedup/DuplicateReportService.cpp:1313`
- `DedupCore/dedup/DuplicateReportService.cpp:1335`
- `DedupCore/dedup/DuplicateReportService.cpp:1429`

### 4.5 当前配置 n 与报告生成时的 n 不一致

报告任务启动时会冻结当时的 `n`，运行中修改配置不会改变正在生成的报告。

另外，当前持久化报告没有完整保存并校验“报告生成时的 n”。因此可能出现：

```text
旧报告按 n=4 生成
当前配置已改为 n=8
某文件对距离为 6
界面继续显示旧报告，因此该文件对仍然不存在
```

这属于旧报告与当前配置语义不一致，不是当前分桶计算错误。

### 4.6 文件或数据库数据在报告生成后发生变化

报告使用生成时 MySQL 中保存的 SHA-512 和 dHash。

如果报告生成后文件被覆盖、替换或重新计算，而界面仍加载旧报告，则“当前重新计算得到的 dHash 距离”和“报告生成时使用的 dHash 距离”可能不同。

### 4.7 运行了旧版本程序或加载了旧报告

Debug、Release 或其他输出目录中可能存在旧的 `VideoScGUI.exe`。旧程序可能仍使用旧分组规则，已生成的旧报告也不会因为程序升级自动变成新规则结果。

### 4.8 报告生成取消或失败

当前候选索引写入、RocksDB 查询或内容解析失败时，报告生成会失败，不会正常发布完整的新 generation。

如果界面继续加载上一次成功报告，就可能表现为“新数据没有筛选出来”。因此需要同时核对：

- 当前 generation ID。
- 报告生成时间。
- 报告任务最终状态。
- 执行日志和失败日志。

## 5. 已确认的产品语义

已确认采用方案 C。以下方案对比保留为设计依据：

### 方案 A：保留当前非重叠严格组

规则：

- 每个内容只出现一次。
- 同组任意两个内容都满足 `distance <= n`。
- 允许部分直接相似文件对分处不同组。

优点：

- 结果组互不重叠。
- 删除选择和可释放空间统计较直接。
- 组内严格相似约束清晰。

缺点：

- 不能完整展示全部直接相似关系。

### 方案 B：允许重叠严格组

规则：

- 同一个内容可以出现在多个组中。
- 每个组内任意两个内容都满足 `distance <= n`。
- 尽量让每条直接相似关系至少被一个组覆盖。

优点：

- 可以保留更多直接相似关系。

缺点：

- 同一文件会重复出现。
- 删除选择、可释放空间和组统计必须全局去重。
- 如果枚举全部最大团，最坏情况下计算和输出规模会非常大。

### 方案 C：严格主分组加相似文件对明细

规则：

- 主页面继续使用非重叠严格组。
- 另外保存或按需查询所有满足 `distance <= n` 的直接相似边。
- 详情窗口提示“该文件还与其他组中的文件相似”。

优点：

- 保留当前主分组、删除和空间统计语义。
- 用户仍能查到没有被同组覆盖的相似文件对。
- 不需要枚举全部最大团。

缺点：

- 报告数据量增加。
- 详情页需要新增跨组相似关系入口。

建议优先采用方案 C。它同时保留“组内任意两项严格相似”和“每一个直接相似文件对都可以被查询”两种能力。

## 6. 已确认的修改方向

方案 C 计划执行以下修改：

1. 在候选真实汉明距离复核通过时，写入规范化的相似边：

```text
edge(min_sha512, max_sha512) -> distance
```

2. 分组阶段继续使用当前非重叠完整链接规则，不改变组内严格不变量。
3. 为每个内容建立跨组相似邻接索引，详情页按需加载。
4. 详情页显示：

```text
同组严格相似成员
其他组中的直接相似成员
SHA-512
dHash
汉明距离
所属组
```

5. 可释放空间和删除选择仍只按主分组计算，跨组相似边不重复计入空间。
6. 报告元数据增加：

```text
image_max_hamming_distance
image_bucket_index_version
similarity_grouping_version
similarity_edge_version
generated_utc_ms
```

7. 配置 `n` 变化后，将旧报告标记为过期并提示重新生成。
8. 日志记录以下计数：

```text
有效图片内容数
候选召回数
真实距离比较数
距离 <= n 的相似边数
被主分组覆盖的相似边数
跨组相似边数
无活动路径的相似内容数
```

## 7. 测试计划

### 7.1 分桶无漏召回测试

对所有允许的 `n`：

- 随机构造大量汉明距离 `0..n` 的 dHash 对。
- 验证至少命中一个相同序号分段。
- 验证候选查询返回另一方。
- 验证同一候选命中多个桶时只返回一次。

### 7.2 链式关系测试

构造：

```text
A-B <= n
B-C <= n
A-C > n
```

验证：

- 主分组不会把 `A、B、C` 全部放入同一组。
- 相似边报告同时包含 `A-B` 和 `B-C`。
- `B-C` 即使跨组，也能在详情页查询到。

### 7.3 多团重叠测试

构造同一内容属于多个合法严格团的图，验证：

- 主分组中内容只出现一次。
- 相似边完整保存。
- 跨组相似关系不会重复计入可释放空间。

### 7.4 数据过滤测试

分别验证：

- dHash 为空。
- dHash 为 0。
- 媒体类型错误。
- 算法版本错误。
- 数据仅存在本地、尚未同步 MySQL。
- 没有活动路径。

界面或日志必须能区分“算法没有匹配”和“数据未进入报告输入”。

### 7.5 报告生命周期测试

- 使用不同 `n` 生成报告后切换配置，旧报告必须显示生成阈值或过期状态。
- 取消和生成失败时不得把不完整 generation 发布为新报告。
- 界面不得把上一次成功报告误显示为本次新结果。

## 8. 验收标准

1. 对所有已进入本次候选索引的有效图片内容，距离 `<= n` 的 dHash 对不会因分桶而漏召回。
2. 主分组内任意两个不同内容的汉明距离始终 `<= n`。
3. 若产品要求保留全部直接相似关系，则每一条距离 `<= n` 的相似边都可在报告或详情页中查询。
4. 数据未同步、dHash 无效、算法版本错误、无活动路径、旧报告等情况均有明确日志或界面提示。
5. 报告显示生成时的 `n`、生成时间、generation ID 和分组规则版本。
6. 同一文件跨组相似时，不重复计算可释放空间，也不会产生相互冲突的删除选择。
