# dHash 内容相似计算进度条修改计划

> 日期：2026-07-16  
> 状态：已完成修改；Debug/Release x64 构建及测试通过

## 1. 修改目标

1. 在重复报告窗口执行“手动计算 dHash 相似报告”或扫描后自动生成 dHash 相似报告时，显示清晰的阶段进度条。
2. 显示当前阶段序号、中文阶段名称、阶段内已处理数量和总量。
3. 有可靠总量的阶段显示真实百分比；没有可靠总量的 MySQL 流式读取阶段显示动画进度和实际累计数。
4. 进度条更新及时，但不因高频回调增加明显的 GUI 锁竞争和报告计算开销。
5. 取消、失败、成功状态使用明确文案，不把取消或失败显示为 100% 完成。
6. 保留现有内容数、路径数、相似边数和分组数等诊断信息。
7. 项目中所有进度条的覆盖文字均使用其所在位置底色的 RGB 反色，确保填充区和未填充区上的文字始终保持对比度。

本计划中的“dHash 内容相似计算”指基于 MySQL 已有图片/视频 dHash 生成相似报告，不重新执行图片 dHash 或视频六帧 dHash 的媒体特征提取。

## 2. 当前实现盘点

### 2.1 当前 GUI 只有文本计数

`VideoScGUI/VideoScApp.cpp` 的重复报告窗口在报告运行时只显示：

```text
报告阶段：<内部阶段键> | 内容 N | 路径 N | 相似边 N | 分组 N
```

当前问题：

1. 没有调用 `ImGui::ProgressBar()`。
2. 阶段直接显示英文内部键，不便于快速理解。
3. 没有阶段序号、阶段总量和百分比。
4. 用户无法判断任务是在正常推进还是已经停滞。

项目中扫描窗口已经有确定型和不确定型 `ImGui::ProgressBar()`，并已有：

- `CalculateStageProgress()`：计算并限制 `[0, 1]` 进度值。
- `FormatStageProgress()`：生成“已处理 P / T（X%）”覆盖文字。

本次直接复用现有 GUI 辅助函数，不新增另一套进度条计算逻辑。

当前项目业务界面中已有的扫描进度条和本计划新增的 dHash 相似进度条都位于
`VideoScGUI/VideoScApp.cpp`。本次同步统一两处进度条文字绘制方式，避免只有新增进度条采用反色文字。

### 2.2 当前进度模型没有阶段分母

`DedupCore/dedup/DuplicateReportService.h` 的 `DuplicateReportProgress` 当前只有：

- `processed_contents`
- `processed_paths`
- `matched_pairs`
- `emitted_groups`
- `stage`

这些字段是跨阶段累计诊断数据，不是稳定的“当前阶段已处理/总量”，不能直接作为进度条分子和分母。

### 2.3 当前 dHash 相似报告阶段

`DuplicateReportGenerator::GenerateSimilar()` 当前包含六个主要阶段：

1. `streaming_mysql_visual_contents`
   - 从 MySQL 流式读取有效图片和视频视觉内容。
   - 写入本次 RocksDB 工作区并建立 dHash 候选索引。
2. `grouping_identical_visual_signatures`
   - 遍历本次视觉内容。
   - 合并完全相同的视觉签名，并建立代表内容。
3. `matching_dhash_candidates`
   - 遍历代表内容。
   - 查询 dHash 候选、执行真实汉明距离复核并生成相似边。
4. `joining_active_paths`
   - 从 MySQL 流式读取 active 路径。
   - 将路径关联到相似内容根节点。
5. `organizing_similarity_evidence`
   - 遍历相似边并整理每个分组的距离证据。
6. `writing_similarity_groups`
   - 组装成员、证据和可释放大小。
   - 写入并发布最终相似报告。

### 2.4 当前进度回调存在的缺口

1. 第二阶段继续使用第一阶段的 `processed_contents`，没有独立的阶段计数。
2. 第二阶段的回调条件依赖一个不会在该阶段变化的累计值，可能出现一直不回调或过度回调。
3. 第三阶段没有“已处理代表内容数”，长时间候选匹配期间界面只显示静态文字。
4. 第五阶段没有“已整理相似边数”。
5. 第六阶段只有成功写入报告组时才增加计数；被检查但成员不足两项的根节点没有计入阶段处理量。
6. 当前固定按 1024 项或 64 组回调；单项耗时较高时，进度界面可能较长时间没有更新。

## 3. 推荐展示方案

重复报告窗口在 dHash 相似报告运行时显示：

```text
当前阶段：3/6  匹配 dHash 候选
[██████████████████----------------------] 已处理 1,248 / 3,920（31.8%）

视觉内容 8,615 | 有效路径 12,306 | 相似边 423 | 已写入分组 118
```

### 3.1 六个阶段的进度规则

| 阶段 | 中文名称 | 进度条类型 | 分子 | 分母 |
| --- | --- | --- | --- | --- |
| 1/6 | 读取 MySQL 视觉内容 | 不确定动画 | 已读取视觉内容数 | 流式读取结束前未知 |
| 2/6 | 合并相同视觉签名 | 确定进度 | 已检查视觉内容数 | 第一阶段实际读取数 |
| 3/6 | 匹配 dHash 候选 | 确定进度 | 已检查代表内容数 | 第二阶段生成的代表内容数 |
| 4/6 | 关联有效文件路径 | 不确定动画 | 已读取 active 路径数 | 流式读取结束前未知 |
| 5/6 | 整理相似度证据 | 确定进度 | 已整理相似边数 | 前面阶段实际写入的相似边数 |
| 6/6 | 写入相似分组 | 确定进度 | 已检查分组根节点数 | 第四阶段建立的分组根节点数 |

不为第一、第四阶段额外执行 MySQL `COUNT(*)`：

- 避免报告开始前增加两次全表统计查询。
- 避免统计查询与后续流式查询之间数据变化造成分母不一致。
- 这两个阶段使用 ImGui 原生不确定动画，并显示持续增长的真实累计数。

### 3.2 不显示伪造的总体百分比

六个阶段的单项成本差异很大：

- 读取一条 MySQL 记录与匹配一组 dHash 候选耗时不可直接等价。
- 候选数量随媒体内容分布变化。
- 相似边和最终分组数量在运行前未知。

因此本次显示“阶段序号 + 当前阶段进度”，不把六个阶段简单平均成总体百分比，避免阶段切换时产生误导性的跳变。

## 4. 进度数据模型调整

修改 `DedupCore/dedup/DuplicateReportService.h`，在 `DuplicateReportProgress` 中增加：

- `stage_index`：当前阶段序号，dHash 相似报告为 `1` 到 `6`。
- `stage_count`：阶段总数，dHash 相似报告固定为 `6`。
- `stage_processed`：当前阶段已经完成检查的项目数。
- `stage_total`：当前阶段总项目数。
- `stage_total_known`：`stage_total` 是否可靠可用。

继续保留现有累计字段：

- `processed_contents`：第一阶段实际读取的视觉内容累计数。
- `processed_paths`：第四阶段实际读取的 active 路径累计数。
- `matched_pairs`：成功保存的相似边累计数。
- `emitted_groups`：最终成功写入的相似组累计数。

阶段切换时只重置 `stage_processed`、`stage_total` 和 `stage_total_known`，不重置累计诊断字段。

## 5. 核心计算流程调整

修改 `DedupCore/dedup/DuplicateReportService.cpp`。

### 5.1 统一阶段切换

在文件内部增加最小的私有辅助逻辑：

1. 设置阶段键、阶段序号和总阶段数。
2. 重置当前阶段已处理数。
3. 设置当前阶段总量是否已知。
4. 阶段切换时强制发布一次进度快照。

该逻辑只服务重复报告生成，不提取为跨模块公共工具。

### 5.2 第一阶段：读取视觉内容

1. 设置 `stage_total_known=false`。
2. 每成功处理一条 MySQL 视觉内容：
   - 增加 `processed_contents`。
   - 增加 `stage_processed`。
3. 第一阶段结束后，将最终 `processed_contents` 作为第二阶段的 `stage_total`。

### 5.3 第二阶段：合并相同视觉签名

1. 每访问一条工作区视觉内容都增加 `stage_processed`。
2. 无效视觉内容、静态视频等提前返回分支也必须计入已检查数量，避免进度停在 99%。
3. 每首次创建一个 `representative/` 记录时增加代表内容总数。
4. 第二阶段结束后，将实际代表内容数作为第三阶段总量。
5. 修正当前使用第一阶段累计值判断回调频率的问题。

### 5.4 第三阶段：匹配 dHash 候选

1. 每完成一个代表内容的候选查询与复核后增加 `stage_processed`。
2. 每成功保存一条新的相似边继续增加 `matched_pairs`。
3. 候选数量不作为分母；分母只使用代表内容总数，保证进度单调且可预测。

### 5.5 第四阶段：关联 active 路径

1. 设置 `stage_total_known=false`。
2. 每读取一条 active 路径增加：
   - `processed_paths`
   - `stage_processed`
3. 每首次创建一个 `count/<root>` 分组计数键时，增加分组根节点总数。
4. 第四阶段结束后，将分组根节点总数作为第六阶段总量。

### 5.6 第五阶段：整理相似度证据

1. 使用前面成功保存的 `matched_pairs` 作为阶段总量。
2. 每成功检查并整理一条 `edge/` 记录后增加 `stage_processed`。
3. 总量为零时按无需处理处理，立即进入下一阶段，不执行除零。

### 5.7 第六阶段：写入相似分组

1. 每检查一个 `count/<root>` 记录都增加 `stage_processed`。
2. 即使成员数量不足两项、最终不写入报告，也计入阶段已处理数。
3. `emitted_groups` 仍只统计真正保存成功的相似组，不能复用为阶段已处理数。
4. 全部成功后发布完成快照，再原子发布报告。

### 5.8 回调节流

将当前按固定数量取模的回调方式调整为时间节流：

- 运行阶段最多约每 `100 ms` 发布一次进度快照。
- 阶段开始、阶段结束、任务完成时强制发布。
- 不在持有 RocksDB 或 MySQL 内部锁之外新增长时间锁。
- GUI 回调继续只在 `m_reportResultMutex` 下复制一个轻量结构。

这样既能让耗时单项及时反馈，也避免每条记录都触发 GUI 同步。

## 6. GUI 调整

修改 `VideoScGUI/VideoScApp.cpp`。

### 6.1 阶段中文名称

增加文件内部阶段名称映射，将内部稳定键转换为：

- 读取 MySQL 视觉内容
- 合并相同视觉签名
- 匹配 dHash 候选
- 关联有效文件路径
- 整理相似度证据

未知阶段键显示“准备 dHash 相似计算”，不能直接显示空字符串。

### 6.2 绘制进度条

仅在当前运行任务为 `DuplicateReportKind::Similar` 时显示新增进度条：

1. 显示 `当前阶段：X/6 <中文阶段名>`。
2. `stage_total_known=true`：
   - 使用 `CalculateStageProgress()`。
   - 使用 `FormatStageProgress()` 生成覆盖文字。
3. `stage_total_known=false`：
   - 使用 `-ImGui::GetTime()` 绘制不确定动画。
   - 覆盖文字显示“已处理 N 项”或“已读取 N 条路径”。
4. 进度条宽度使用 `ImGui::GetContentRegionAvail().x`，跟随窗口宽度。
5. 下方保留现有累计内容、路径、相似边和分组统计。

SHA-512 精确报告保持现有展示，不在本次范围内增加阶段百分比。

### 6.3 取消状态

点击“取消报告生成”后：

1. 禁用重复点击取消按钮。
2. 进度条停止不确定动画，保留最后一次进度位置。
3. 覆盖文字显示“正在取消，等待当前操作安全退出……”。
4. 后台任务退出后沿用现有“报告生成已取消，未发布半成品”提示。
5. 不把取消状态强制绘制为 100%。

自动从 SHA-512 精确报告切换到 dHash 相似报告时，继续使用现有 `m_reportProgress = {}` 清空上一任务进度，避免继承前一个报告的阶段和计数。

### 6.4 项目进度条文字反色

项目内所有业务进度条统一使用一个文件内部绘制辅助函数，不再直接依赖
`ImGui::ProgressBar()` 的默认覆盖文字颜色。

#### 反色规则

反色按当前实际底色逐通道计算：

```text
text.r = 1.0 - background.r
text.g = 1.0 - background.g
text.b = 1.0 - background.b
text.a = 1.0
```

- 填充区域文字使用 `ImGuiCol_PlotHistogram` 当前颜色的反色。
- 未填充区域文字使用 `ImGuiCol_FrameBg` 当前颜色的反色。
- 颜色必须从当前 ImGui Style 读取，不能写死为黑色或白色，以便主题或阶段颜色变化后仍保持正确反色。

#### 跨越填充边界的文字

同一段覆盖文字可能同时位于填充区和未填充区，不能只为整段文字设置一种颜色。统一绘制函数按以下方式处理：

1. 先调用 `ImGui::ProgressBar()` 绘制进度条本体，但不让其绘制默认覆盖文字。
2. 获取进度条实际矩形和文字居中位置。
3. 用未填充底色的反色绘制一遍完整文字，并把裁剪区域限制在未填充区域。
4. 用填充底色的反色再次绘制同一段文字，并把裁剪区域限制在填充区域。
5. 确定型进度条的填充区域为左侧连续区域。
6. 不确定动画进度条的填充区域为移动的中间色块，左右未填充区域分别裁剪绘制。

这样文字跨越进度边界时，每个字符像素都使用其实际底色的反色，不会出现一半清晰、一半融入背景的情况。

#### 统一覆盖范围

本次至少统一：

1. 扫描窗口现有的扫描阶段进度条。
2. 本计划新增的 dHash 内容相似计算进度条。

后续项目新增业务进度条时必须复用同一辅助函数。`third_party/imgui` 中的官方 Demo 进度条不属于应用功能界面，不纳入修改范围。

## 9. 验收标准

1. 手动或自动启动 dHash 相似报告后立即显示进度条。
2. 界面显示六个中文阶段及正确阶段序号。
3. 第二、第三、第五、第六阶段显示真实“已处理/总数”和百分比。
4. 第一、第四阶段显示动画进度及持续增长的实际累计数，不显示虚假百分比。
5. 确定型进度不倒退、不超过 100%，总量为零时不除零。
6. 无效视觉内容、成员不足两项的分组根节点仍计入阶段已处理量，不会停在 99%。
7. 大量候选匹配期间界面至少约每 `100 ms` 有机会获得新快照，不因固定 1024 项门槛长期不更新。
8. 点击取消后进度动画停止，最终提示取消且不发布半成品。
9. 报告最终分组数量、dHash 判定结果和已发布报告内容与修改前一致。
10. 扫描进度条和 dHash 相似进度条的文字在填充区域使用填充色反色，在未填充区域使用背景色反色。
11. 覆盖文字跨越填充边界时，两侧分别使用对应底色反色；不确定动画色块移动时文字颜色裁剪同步移动。
12. 切换阶段颜色、警告色、错误色和成功色后，文字反色根据实际 Style 颜色实时计算，不使用固定黑白色。
13. Debug/Release x64 构建成功，现有 `DedupTests` 全部通过。
