# 重复组详情卡顿与图片预览不可用修复计划

> 日期：2026-07-16  
> 状态：已执行

## 1. 问题现象

1. 打开重复组详情窗口后，界面明显卡顿，组内成员或相似度证据越多越严重。
2. 图片重复组的预览列显示“预览不可用”。
3. 当前失败日志只能看到 `preview_unavailable`，无法判断失败发生在：
   - 原文件打开。
   - WIC 解码。
   - 缩放。
   - 临时 PNG 编码。
   - D3D11 纹理创建。

## 2. 根因结论

### 2.1 详情成员表没有真正虚拟化

`VideoScApp::RenderDuplicateReportDetailWindow()` 为成员表设置了 `ScrollY`，但仍然在每一帧执行：

```cpp
for (const auto& member : group.members)
```

因此滚动区域之外的成员仍会执行完整业务逻辑，包括：

- 文件存在性检查。
- 图片解码与缩放。
- 临时 PNG 写入。
- D3D11 纹理创建。
- SHA-512、dHash 和其他 ImGui 控件创建。

表格有滚动条不等于业务代码自动停止处理不可见行。当前没有使用 `ImGuiListClipper`，也没有检查行是否实际可见。

### 2.2 图片预览全部在主 UI 线程同步执行

`AcquireReportThumbnail()` 直接在 ImGui 渲染线程调用 `LoadImageToTexture()`。首次打开详情时，每一张未缓存图片依次执行：

1. 创建 WIC Factory。
2. 打开并解码原始媒体文件。
3. 按 `image_preview_long_edge` 缩放。
4. 把像素复制到 CPU 内存。
5. 把缩放结果编码为临时 PNG 文件。
6. 创建 D3D11 Texture 和 Shader Resource View。

上述操作全部发生在单帧内。原图较大、文件位于慢盘或组内成员较多时，窗口会长时间无响应。

此前计划要求“后台生成图片临时缩略图”，但当前实际实现只有视频缺失拼图使用 `std::async`；图片仍为同步实现，未达到原计划目标。

### 2.3 相似度证据表存在重复线性查找

详情窗口每帧遍历全部 `group.evidence`。每条证据显示成员名称时，又分别在全部成员中执行一次 `std::find_if`：

```text
每帧复杂度约为 O(证据边数 × 成员数)
```

同时证据表没有限定滚动高度，也没有使用 `ImGuiListClipper`。大型 dHash 相似组会在缩略图之外继续产生明显 CPU 开销。

### 2.4 相同 SHA-512 内容没有复用预览

图片预览缓存当前以原始文件路径作为键。SHA-512 精确重复组的多个成员内容完全相同，但仍会：

- 分别读取每个路径。
- 分别生成临时 PNG。
- 分别创建相同的 GPU 纹理。

当前测试媒体中的精确图片重复组为：

```text
E:\Code\media\1.webp
E:\Code\media\generated_test_media\duplicates\image_exact_copy.webp
```

两个文件 SHA-512 完全一致，但当前会被当作两个独立预览任务。

### 2.5 图片预览依赖 WIC 完成全部格式处理

项目的图片 dHash 使用 FFmpeg 解码，能够处理 WebP 等测试格式；详情预览却改用 WIC 直接打开原始文件。两条解码链路支持的格式和错误行为不一致：

- 扫描阶段可以成功计算 dHash。
- 详情阶段仍可能因 WIC 解码器、像素转换器或编码器限制显示“预览不可用”。

当前临时 PNG 写入还要求 WIC PNG 编码器接受精确的
`GUID_WICPixelFormat32bppRGBA`。如果编码器协商为其他兼容像素格式，代码会主动返回
`WINCODEC_ERR_UNSUPPORTEDPIXELFORMAT`，并丢弃已经成功解码的图像。

### 2.6 失败日志缺少真实错误

`AcquireReportThumbnail()` 把所有失败统一写成：

```text
operation=image_thumbnail_decode
detail=preview_unavailable
```

没有记录 HRESULT、原生错误码和实际失败步骤，无法区分文件不存在、格式不支持、临时目录不可写或显存创建失败。

## 3. 修复目标

1. 打开重复组详情窗口时不再同步处理整组全部图片。
2. 只为当前可见及少量预加载成员请求缩略图。
3. 图片解码、缩放和临时文件写入全部移出 ImGui 主线程。
4. D3D11 纹理仍只在主渲染线程创建，但每帧限制上传数量。
5. SHA-512 相同的图片共享同一个临时缩略图和 GPU 纹理。
6. JPEG、PNG、WebP、BMP、TIFF、GIF 等项目已支持图片均可生成详情预览。
7. 大型成员表和相似度证据表使用真实虚拟化。
8. 预览失败时显示可理解的原因，并把真实失败阶段写入执行失败日志。

### 3.1 明确保留图片源资源生成缩略图的功能

重复组详情中的图片预览必须继续采用“原始图片资源生成临时缩略图，再加载缩略图作为预览”的流程，不允许为了修复卡顿而退化为直接加载原始图片。

具体约束如下：

1. 当 `member.media_kind == MediaKind::Image` 时，以 `member.path` 指向的原始图片作为缩略图生成输入。
2. `member.thumbnail_path` 为空、等于原始图片路径或文件不存在时，不视为最终“预览不可用”，而是进入图片缩略图生成队列。
3. 后台任务按照 `image_preview_long_edge` 对原图等比例缩放，并把结果保存到本次运行的临时预览目录。
4. 详情窗口只加载已经生成的有界缩略图来创建 D3D11 纹理，不在 UI 线程直接解码、上传原始大图。
5. 缩略图生成期间显示“正在生成图片预览…”，只有明确失败后才显示对应失败原因。
6. 缩略图不得覆盖或修改原始图片，不写入 MySQL，也不改变重复报告中的 `thumbnail_path`、SHA-512 或 dHash 数据。
7. SHA-512 相同的图片共享一份临时缩略图；SHA-512 不同的相似图片分别生成各自的缩略图。
8. 应用关闭时统一删除本次运行产生的图片临时缩略图。

## 4. 推荐修复方案

### 4.1 成员表使用固定行高虚拟化

成员表继续使用当前 `166` 像素固定行高，在表格内部增加 `ImGuiListClipper`：

1. `clipper.Begin(group.members.size(), 166.0f)`。
2. 只循环 `DisplayStart` 到 `DisplayEnd`。
3. 仅可见成员创建 ImGui 控件并请求预览。
4. 可在可见范围前后各预取一项，但不能一次请求整个组。

这样打开窗口时，工作量只与可见行数有关，而不是与组内总成员数有关。

### 4.2 相似度证据表虚拟化

1. 为证据表设置有界滚动高度。
2. 使用单行固定高度的 `ImGuiListClipper`。
3. 选中重复组时一次性建立：

```text
path_id -> 文件名
```

映射。
4. 渲染证据时执行 O(1) 查找，不再为每条边扫描全部成员。
5. 平均汉明距离只在组切换时计算一次，或复用预先计算的详情状态。

### 4.3 新增有界图片预览后台任务

在 GUI 中增加独立图片预览状态：

```text
未请求 -> 排队 -> 正在生成 -> 临时文件就绪 -> 纹理就绪
                               \-> 失败
```

后台任务负责：

- 读取原始图片。
- 解码和缩放。
- 写入本次运行临时目录。
- 返回输出路径、尺寸和详细错误。

后台并发必须有上限，避免像当前视频 `std::async` 一样随可见项无限创建线程。建议同时最多执行 `2` 个图片预览任务，其余请求进入队列。

后台线程不能调用 ImGui 或直接创建、释放 D3D11 纹理。

### 4.4 使用 FFmpeg 生成标准临时缩略图

为避免扫描与详情使用两套格式能力，在 `VideoSc` DLL 增加单一职责的图片预览接口：

```text
GenerateImagePreview(
    source_path,
    output_path,
    maximum_long_edge,
    thread_count,
    timeout
)
```

该接口复用项目已有：

- libavformat/libavcodec 图片首帧解码。
- libswscale 尺寸缩放和 RGB 转换。
- `SaveImageFile()` 输出 PNG/JPEG。
- DLL 异常边界、超时和取消处理。

GUI 后台线程调用该接口生成标准 PNG 临时文件。主线程只用 WIC 读取已经有界的标准 PNG 并创建 D3D11 纹理。

不再让 WIC直接处理任意原始图片格式，也不再在主线程执行临时 PNG 编码。

### 4.5 按内容摘要复用预览

图片预览的稳定缓存键改为：

```text
SHA-512 十六进制 + 图片预览长边 + 输出格式
```

效果：

- SHA-512 精确重复组只生成一次缩略图。
- 同一内容存在多个路径时共享一个临时文件和 GPU 纹理。
- 配置中的图片预览尺寸或输出格式变化后生成不同缓存项。
- 不使用路径作为唯一键，避免相同内容重复解码。

视频仍按现有拼图路径处理；历史视频按需生成任务继续使用路径区分。

### 4.6 主线程限量上传纹理

每帧开始或结束时收取已完成后台任务：

1. 每帧最多把 `1～2` 个临时缩略图上传为 D3D11 纹理。
2. 只有当前仍可见或仍存在缓存需求的结果才上传。
3. 超出 `cache_entries` 或 `gpu_memory_limit_mib` 时保持为临时文件就绪状态，等待缓存释放后再上传。
4. 纹理创建失败不能重新解码原图，只记录 D3D11 阶段失败。

### 4.7 缓存与生命周期

1. 临时图片仍保存在：

```text
%TEMP%\VideoScGUI\preview-<process-id>-<run-id>
```

2. 正常关闭时：
   - 停止接受新任务。
   - 等待或取消后台预览任务。
   - 释放 GPU 纹理。
   - 删除本次运行临时目录。
3. 异常退出遗留目录继续由下次启动清理。
4. 切换重复组时不必删除同 SHA-512 的临时文件，可复用到本次运行结束。
5. `ClearReportThumbnails()` 只释放 GPU 纹理，不清除已经完成的临时文件索引；关闭应用时再统一删除临时目录。

## 5. 失败展示与日志

### 5.1 界面状态

预览列区分显示：

- `等待生成预览…`
- `正在生成预览…`
- `原文件不存在`
- `图片解码失败`
- `临时缩略图写入失败`
- `纹理创建失败`
- `预览缓存已满`

不再把所有状态统一显示为“预览不可用”。

### 5.2 日志字段

执行失败日志至少记录：

- `path_id`
- 原始路径
- `storage_target_key`
- SHA-512
- 媒体类型
- 失败阶段
- DLL 状态码
- Windows/FFmpeg 原生错误码
- 临时输出路径
- 详细错误文本

建议使用以下稳定操作名：

```text
image_preview_source_open
image_preview_decode
image_preview_scale
image_preview_write
image_preview_wic_load
image_preview_d3d_upload
```

## 6. 预计修改文件

| 文件 | 计划修改 |
| --- | --- |
| `VideoSc/VideoSc.h` | 增加图片预览生成选项、结果和稳定 DLL API |
| `VideoSc/dllmain.cpp` | 使用 FFmpeg 解码、缩放并保存有界图片预览，补齐取消、超时与异常边界 |
| `VideoScGUI/VideoScApp.h` | 增加图片预览键、任务状态、结果队列、内容缓存和证据成员名索引 |
| `VideoScGUI/VideoScApp.cpp` | 虚拟化成员/证据表，后台请求图片预览，限量上传纹理，细化失败展示和日志 |
| `DedupTests/main.cpp` | 增加多格式图片预览、尺寸限制、失败安全和相同 SHA-512 复用键测试 |

不修改：

- SHA-512 算法。
- 图片 dHash、视频六帧 dHash算法。
- dHash 相似阈值和分组结果。
- 重复报告 RocksDB 格式。
- MySQL schema。
- 媒体文件永久删除流程。

## 7. 实施步骤

1. 在 `VideoSc` DLL 中提取可复用的图片首帧解码与有界 RGB 缩放逻辑。
2. 增加 `GenerateImagePreview()` 公共 API，并保持所有异常不跨 DLL 边界。
3. 为 JPEG、PNG、WebP、BMP、TIFF、GIF 增加预览生成测试。
4. 在 GUI 中增加以 SHA-512、预览尺寸和格式组成的内容缓存键。
5. 增加最多两个并发任务的图片预览队列。
6. 把临时缩略图生成放入后台任务，移除 `AcquireReportThumbnail()` 中的原图同步缩放和 PNG 编码。
7. 修改详情窗口的图片分支：原始图片路径只用于提交缩略图生成任务，生成完成前不得传给纹理加载函数。
8. 主线程按每帧固定上限收取完成结果，并且只从生成后的临时缩略图创建 D3D11 纹理。
9. 成员详情表接入 `ImGuiListClipper`，只处理可见成员。
10. 相似度证据表增加有界滚动和 `ImGuiListClipper`。
11. 选中组时预建 `path_id -> 文件名` 映射，消除证据表重复线性查找。
12. 细分预览失败状态和日志字段。
13. 完善关闭、切组、缓存淘汰和异常退出清理流程。
14. 执行 Debug/Release x64 构建及全部 `DedupTests`。
15. 使用真实 WebP 精确重复组和大成员组进行 GUI 人工验收。

## 8. 验收标准

1. 打开包含大量成员的重复组详情时，主窗口仍可拖动、停靠和响应点击。
2. 首帧不再同步解码整组图片，只请求当前可见成员。
3. 快速滚动时，预览按可见位置异步加载，不出现长时间界面冻结。
4. 相似度证据较多时，详情窗口每帧不再执行 `证据边数 × 成员数` 的线性查找。
5. 以下两个 SHA-512 相同的 WebP 文件均能显示预览：

```text
E:\Code\media\1.webp
E:\Code\media\generated_test_media\duplicates\image_exact_copy.webp
```

6. 上述两个文件在同一次运行中只生成一份临时缩略图并共享 GPU 纹理。
7. JPEG、PNG、WebP、BMP、TIFF、GIF 预览生成成功；坏图安全失败且不崩溃。
8. 临时缩略图长边不超过配置的 `image_preview_long_edge`。
9. 预览生成不改变原始媒体文件，也不重新计算或覆盖报告中的 SHA-512/dHash。
10. 关闭应用后，本次运行的图片临时缩略图目录被删除。
11. 缓存条目或显存达到上限时界面仍保持响应，并显示明确状态。
12. 每种预览失败均能在执行失败日志中看到具体阶段、路径和错误码。
13. Debug/Release x64 构建成功，全部现有及新增测试通过。
14. 图片成员打开详情后必须先生成或复用临时缩略图，并仅使用该缩略图显示预览；不得直接把原始图片作为详情纹理来源。

## 9. 本次不在范围内

1. 不在扫描阶段永久保存图片缩略图。
2. 不把图片缩略图同步到 MySQL。
3. 不修改重复报告生成和分组算法。
4. 不预先为全部报告或全部成员生成缩略图。
5. 不在后台线程直接访问 ImGui 或 D3D11 纹理对象。

## 10. 执行结果

1. 已增加 `GenerateImagePreview()` DLL 接口，使用 FFmpeg 解码图片首帧并生成有界 PNG/JPEG 临时缩略图。
2. 图片详情预览已改为最多两个后台任务并发，UI 线程不再同步解码、缩放和编码原始图片。
3. 图片预览缓存已改为以 SHA-512、预览最长边和输出格式组成内容键，精确重复图片共享临时缩略图和 GPU 纹理。
4. 主线程每帧最多上传两个报告预览纹理。
5. 成员表和相似度证据表已接入 `ImGuiListClipper`，证据成员名称改为预构建索引查询。
6. 图片缩略图生成失败会记录 DLL 状态码、原生错误码和详细错误文本。
7. 已增加 WebP 图片缩略图生成、尺寸限制和无效参数测试。
8. Debug x64 与 Release x64 均构建成功。
9. `DedupTests` 共 36 项全部通过。
