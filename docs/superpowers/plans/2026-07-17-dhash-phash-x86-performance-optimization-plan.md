# dHash + pHash 二次确认与 x86 性能优化计划

> 日期：2026-07-17  
> 状态：已合并进 `2026-07-17-main-content-image-three-stage-performance-modification-plan.md`，禁止按本文执行代码修改  
> 目标流程：`dHash 分桶 → 长宽比剪枝 → dHash 汉明距离 → pHash 二次确认`  
> 目标平台：保持 x64 发布，针对 x86-64/AMD64 处理器优化

> 取代原因：新需求要求在大量水印、不同分辨率和不同色调下召回同一底图，并明确拒绝“主内容相似但不同”的图片。dHash 作为硬门槛会让被水印扰动的图片无法进入 pHash 二筛；新修改方案已将 `PDQ-256 → 分区 pHash → 归一化结构直验` 与本文可复用的 x64/CPU 优化合并，并补齐索引、缓存和迁移设计。本文只保留为历史参考。

## 1. 当前源码事实

1. 图片使用一个 64 位 dHash；视频使用六个固定采样帧的 64 位 dHash。
2. 图片 dHash 候选索引已经使用 `n+1` 段 MIH 分桶，可以保证汉明距离 `<= n` 的对象至少共享一个分段。
3. 候选对在 `DuplicateReportService` 的独立报告线程池中执行长宽比、时长、dHash 距离和严格完整链接校验。
4. 64 位汉明距离当前直接调用 `__popcnt64`，没有在不支持 POPCNT 的处理器上提供兼容分支，也没有批量 SIMD 路径。
5. 媒体 DLL 已经在同一次图片解码或视频六帧解码中生成 dHash，适合在不重复解码的前提下追加 pHash。
6. `ShaFileData`、RocksDB 编码、MySQL `sha512_file_data`、同步消息和重复报告格式目前都没有 pHash 字段。
7. 解决方案虽然存在 `Debug|x86`、`Release|x86` 名称，但 `VideoScGUI`、`DedupCore`、`DedupTests` 等项目仍映射到 x64；当前完整交付链路实际是 x64。
8. 当前 MySQL 模式版本为 1，既有结构变更必须通过备份后的显式迁移，不能只修改 `CREATE TABLE IF NOT EXISTS`。

## 2. 推荐实施口径

### 2.1 推荐目标平台

推荐把“x86 架构”解释为 **x86-64/AMD64 处理器，应用继续发布 x64**：

- 保持现有 `x64-windows` vcpkg 依赖和 x64 构建产物；
- 基线实现不要求 AVX2，保证普通 x86-64 CPU 可以运行；
- 启动时检测 CPU 能力，按 `Scalar/SSE2 → POPCNT → AVX2` 选择实现；
- 不对整个项目设置全局 `/arch:AVX2`，避免程序在旧 CPU 启动即触发非法指令；
- 只把独立的 AVX2 pHash 内核源文件按 `/arch:AVX2` 编译，并由运行时分派器调用。

如果用户要求真正的 **32 位 Win32/x86 可执行文件**，需要额外完成：

- 为 `DedupCore`、`VideoScGUI`、`DedupTests`、崩溃报告器增加 Win32 项目配置；
- 安装并验证 `x86-windows` 的 FFmpeg、RocksDB、MySQL、OpenSSL 等全部依赖；
- 检查 2～3 GiB 用户地址空间下的大报告缓存、RocksDB block cache、FFmpeg 缓冲和线程栈；
- 修正当前解决方案中“x86 名称映射到 x64”的配置；
- 分别交付 x86/x64 二进制和依赖目录。

该分支工作量和风险显著更高，不能与“只针对 x86-64 CPU 优化”混为同一个验收项。

### 2.2 pHash 算法版本

建议定义稳定版本 `phash-dct64-v1`：

1. 从已经解码的原始帧转换为固定 32×32 `GRAY8`；
2. 使用预计算系数执行可分离二维 DCT，只计算最终需要的 8×8 低频区域；
3. 排除 DC 系数后，对其余 63 个低频系数取中位数；
4. 按固定顺序生成 64 位结果，其中保留一个明确约定的占位位，避免不同实现产生位序差异；
5. 标量、SSE2 和 AVX2 实现必须逐样本输出完全相同的 64 位值；
6. 算法版本写入媒体数据、同步消息和报告元数据，旧算法数据不得按新版本直接比较。

为保证跨 CPU 确定性，推荐使用定点 DCT或禁止 FMA 改变舍入结果；不能让 AVX2/FMA 与标量实现偶发产生不同哈希。

## 3. 目标数据流

### 3.1 新媒体计算

```text
读取文件
  → 解码图片首帧 / 视频六个固定采样帧
  → 同一解码帧分别缩放 9×8 和 32×32 灰度图
  → 生成 dHash
  → 生成 pHash
  → 原子写入本地 ShaFileData
  → 按现有同步批量写入 MySQL
```

约束：

- 不得为了 pHash 再次打开和解码同一个媒体文件；
- 每个媒体工作线程复用对齐的 32×32灰度缓冲和 DCT 临时缓冲；
- pHash 失败不覆盖已经成功的 SHA-512/dHash，但必须记录独立失败原因；
- 视频任意一帧 pHash 为无效值时，整组六帧 pHash 视为不可用于二次确认。

### 3.2 历史数据补算

推荐采用“新数据直接计算、历史数据按候选惰性补算”的方式：

1. dHash 分桶先生成候选对；
2. 完成长宽比、时长和 dHash 距离剪枝；
3. 收集剩余候选对涉及的唯一 SHA-512 内容；
4. 从 RocksDB/MySQL 读取已有 pHash；
5. 仅对缺失 pHash 且存在本地可读路径的内容补算一次；
6. 立即持久化并进入同步队列，后续报告复用；
7. 没有本地可读副本的内容按已确认策略处理，不能静默伪造 pHash。

这样避免首次升级后对全部历史媒体执行一次昂贵的全量 pHash 重算。

### 3.3 报告候选与严格分组

建议最终流程固定为：

```text
阶段 1：dHash MIH 分桶枚举候选
阶段 2：图片长宽比 / 视频时长剪枝
阶段 3：dHash 汉明距离校验
阶段 4：识别“接近阈值/疑似误判”的 pHash 触发候选
阶段 5：只补齐触发候选涉及内容的 pHash
阶段 6：只对触发候选执行 pHash 汉明距离二次确认
阶段 7：按条件边判定执行严格完整链接分组
阶段 8：写入报告、距离证据和算法元数据
```

已确认 dHash 是默认大规模初筛；pHash 不覆盖全部 dHash 候选，只处理接近阈值或疑似误判的候选。建议条件边语义如下，等待触发范围最终确认：

- 所有图片成员对必须满足 `dHash距离 <= 图片dHash阈值`；
- 所有视频成员对必须满足 `六帧dHash平均距离 <= 视频dHash阈值`；
- 被标记为 pHash 触发候选的图片边，额外要求 `pHash距离 <= 图片pHash阈值`；
- 被标记为 pHash 触发候选的视频边，额外要求 `六帧pHash平均距离 <= 视频pHash阈值`；
- 未触发 pHash 的边是否仅凭 dHash 通过，仍需用户最终确认；
- 最终分组继续对“包含条件 pHash 规则的边判定”执行严格完整链接，不能只相对某个基准成员通过；
- pHash 只做 dHash 候选的二次确认，不建立第二套全量分桶索引；
- 触发 pHash 的删除安全选择同时保存 dHash/pHash 实测距离、触发原因和冻结阈值，删除前重新复核。

## 4. 数据模型与兼容升级

### 4.1 内存与 RocksDB 模型

在媒体内容模型中增加：

- 图片：`optional<uint64_t> image_phash`；
- 视频：`array<uint64_t, 6> video_phashes`；
- 视频有效标记：`has_video_phashes`；
- 独立 pHash 算法版本，或把现有媒体算法版本提升为包含 dHash/pHash 组合语义的版本。

必须同步修改：

- `CoreModelCodec` 二进制版本；
- `SyncOperation` JSON 编解码；
- 扫描恢复和历史待同步兼容读取；
- 重复报告成员、边证据和报告元数据格式；
- 详情页的十六进制 pHash 显示。

旧 RocksDB 记录缺少 pHash 时必须解码为“缺失”，不能当作零值；新旧记录需要显式版本区分。

### 4.2 MySQL 模式版本 2

建议在 `sha512_file_data` 增加：

- `image_phash BIGINT UNSIGNED NULL`；
- `video_phashes BINARY(48) NULL`；
- `has_video_phashes TINYINT(1) NOT NULL DEFAULT 0`；
- 必要时独立 `phash_algorithm_version VARCHAR(128)`。

实施要求：

1. 初始化器识别 schema version 1；
2. 先调用现有 mysqldump 生成非空备份；
3. 在事务可覆盖的范围内执行 v1 → v2 迁移；
4. 迁移成功后写入 schema version 2；
5. 迁移失败保留备份并停止扫描/同步，禁止部分字段上线；
6. MySQL 读仓库、批量 upsert 和条件同步全部支持新字段。

## 5. x86-64 计算优化

### 5.1 CPU 能力分派

新增只读 CPU 特性快照，在进程首次使用时通过 CPUID 初始化一次：

- SSE2：x64 基线；
- SSSE3/SSE4.1：可选的灰度和中位数辅助；
- POPCNT：64 位汉明距离；
- AVX2：批量像素/DCT 内核；
- FMA：默认不参与会改变哈希结果的计算，除非证明与标量位级一致。

分派器只暴露小接口：

- `ComputePHash64(gray32x32)`；
- `HammingDistance64(left, right)`；
- 可选 `BatchHammingDistance64(query, candidates)`。

### 5.2 pHash 热路径

1. DCT 系数表只初始化一次，按 32 字节对齐。
2. 使用可分离 DCT，并且只计算 8×8 低频输出，不计算无用的完整 32×32 频谱。
3. 每个线程持有独立 scratch buffer，禁止候选循环内重复分配。
4. AVX2 内核一次处理 16 个 `int16` 或 8 个 `float`，但必须与标量结果完全一致。
5. 63 个系数的中位数使用固定容量栈数组和 `nth_element`，不创建动态容器。
6. 内部始终保存原生 `uint64_t`，只在 UI/日志边界转十六进制。

### 5.3 汉明距离热路径

- 单对比较：支持 POPCNT 时使用 `__popcnt64`；否则使用确定性的 SWAR 标量回退。
- 大批候选：根据基准测试决定是否增加 AVX2 nibble lookup 批处理；候选量不足时继续使用 POPCNT，避免 SIMD 装载开销反而变慢。
- 不建议为单个 64 位 pHash强行使用 AVX2；优化重点是批量候选和 DCT。

### 5.4 线程与缓存

- 新扫描的 pHash 归属现有“媒体计算线程池”，与 dHash 共用解码结果。
- 报告中的历史 pHash 补算归属现有“dHash 报告线程池”的独立阶段，不新增第四个长期线程池。
- 同一 SHA-512 在一个报告 generation 内只允许一个补算任务，其他候选复用 future/结果。
- 线程数继续使用配置页的报告校验线程数；如实测 DCT 阶段与距离校验最优并发差异明显，再增加独立 pHash 并发上限。
- 队列保持有界，避免数百万候选把 pHash 任务和内容对象全部放入内存。

## 7. 进度、取消和日志

报告进度增加明确阶段：

1. 枚举 dHash 分桶候选；
2. 长宽比/时长剪枝；
3. dHash 距离校验；
4. 识别 pHash 触发候选；
5. 加载/补算触发候选 pHash；
6. pHash 二次确认；
7. 严格分组；
8. 发布报告。

每阶段显示：

- 输入候选数；
- 被长宽比、dHash、pHash 分别拒绝的数量；
- pHash 缓存命中数、补算数、缺失且不可计算数；

- 活动线程数和已处理数量。

取消时停止领取新内容，当前媒体完成或在 FFmpeg 取消点退出；未发布报告 generation 和临时 pHash 任务索引必须清理，已经完整计算并持久化的 pHash 可以保留复用。

## 8. 自动化测试

### 8.1 算法正确性

1. 固定灰度矩阵的 pHash golden value。
2. 标量、SSE2、AVX2 输出逐样本完全一致。
3. 图片压缩、缩放、亮度变化、水印、裁剪测试集的距离分布。
4. 视频六帧任意零值或缺失值必须拒绝。
5. dHash 通过但 pHash 拒绝的候选不得进入边集合。
6. 最终组任意两成员都满足 dHash/pHash 双阈值。
7. 取消后不发布半成品报告。

### 8.2 数据兼容

1. 旧 RocksDB 媒体记录解码为“pHash 缺失”。
2. 新记录和待同步消息重启后完整恢复。
3. MySQL v1 备份后迁移到 v2。
4. 迁移失败不会写入 schema version 2。
5. 旧 dHash 报告提示重新生成，不能伪装为包含 pHash 的报告。

### 8.3 性能基准

建立独立 Release x64 benchmark，至少记录：

- 纯 32×32 pHash：标量/AVX2 每秒处理量；
- 100 万、1000 万个 64 位汉明距离：SWAR/POPCNT/AVX2 批量吞吐；
- 已解码图片同时生成 dHash+pHash 的增量耗时；
- 图片文件完整读取、解码、双哈希的端到端耗时；
- 视频六帧 dHash 与 dHash+pHash 的端到端差异；
- 报告每阶段候选数量和总耗时。

建议验收目标：

- AVX2 pHash 内核相对标量至少提升 2 倍，否则保留更简单的标量/POPCNT 路径；
- 开启 pHash 后，新媒体端到端吞吐下降不超过实际测试确认的预算；
- 二次确认显著减少 dHash 假阳性，且候选召回测试集没有不可接受的漏检；
- 内核选择不影响哈希结果和最终分组。

## 10. 已确认需求

1. 继续发布 x64，针对 x86-64/AMD64 CPU 做运行时 SIMD 优化，不增加真正的 32 位 Win32 交付。
2. pHash 同时用于图片和视频；图片保存一个 pHash，视频保存六帧 pHash。
3. 默认继续使用 dHash 做大规模初筛，只对距离接近阈值、疑似误判的候选使用 pHash 二次确认。
4. 初始默认值采用图片 `pHash <= 10`、视频六帧平均 `pHash <= 10`，配置页面允许调整，后续以测试媒体标定结果为依据。
5. 新数据在现有媒体解码过程中直接生成 pHash；历史数据只对 dHash 筛选后的 pHash 触发候选惰性补算，并持久化到 RocksDB/MySQL。
6. 必须使用 pHash 的历史候选如果没有 pHash 且没有本地可读副本，则从相似报告排除并写入详细失败日志，不退化为仅 dHash 结果。

## 11. 剩余执行前确认项

### 11.1 “接近 dHash 阈值”的量化方式

推荐采用可配置差值：

```text
图片触发 pHash：图片dHash距离 >= 图片dHash最大距离 - 2
视频触发 pHash：六帧dHash平均距离 >= 视频dHash最大平均距离 - 2
```

- 差值默认 `2`，配置范围 `0～63`；
- 同时仍要求 dHash 距离不超过对应最大值；
- 当最大值小于差值时，下界按 0 处理，避免无符号下溢。

需确认：是否采用该“距离阈值减 2”规则，还是使用百分比或直接配置绝对触发下界？

### 11.2 “疑似误判”的客观条件

推荐不使用不可复现的主观判断，而定义为以下任一条件：

1. 同一个 dHash 候选签名展开出的有效路径数达到配置阈值，默认 `32`；
2. 同一个严格完整链接候选组件在 pHash 前达到配置阈值，默认 `32`；
3. dHash 完全相同但分辨率面积比超过 `4` 倍；长宽比仍必须通过现有容差。

需确认：是否三项全部启用；候选数量默认 `32`、面积比默认 `4` 倍是否接受？

### 11.3 未触发 pHash 的边

推荐规则：未触发 pHash 的候选边只要通过长宽比/时长和 dHash 阈值，就视为相似边；触发 pHash 的边必须额外通过 pHash，否则拒绝。最终对这个条件边集合继续执行严格完整链接。

需确认：是否接受该规则？
