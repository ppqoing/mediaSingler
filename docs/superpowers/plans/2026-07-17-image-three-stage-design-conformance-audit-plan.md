# 图片三级相似设计符合性排查计划

> 日期：2026-07-17  
> 状态：已完成，仅做只读源码与构建配置排查，未修改实现  
> 对照设计：`2026-07-17-main-content-image-three-stage-performance-modification-plan.md`

## 1. 排查目标

确认当前项目的图片相似实现是否完整遵守以下核心边界：

1. 图片必须经过 `PDQ-256 → 4×4 分区 pHash → 结构直验`，不得回退旧 dHash 放行。
2. 仅筛选同一底图的水印、分辨率、编码和色调变体，不支持旋转、裁剪、局部重复或语义相似。
3. 扫描只解码一次，历史缺失特征必须可恢复回填。
4. 特征、配置、同步、MySQL、报告证据和删除选择必须按版本隔离。
5. PDQ 候选不得退化为逐图片大量随机 RocksDB 查询；二、三筛必须有有界并发与缓存。
6. 严格完整链接必须保证组内任意两个不同内容都存在已通过三筛的直接关系。

## 2. 排查范围

- `VideoSc`：解码、PDQ、分区 pHash、结构面、ABI 和 CPU 路径。
- `DedupCore/orchestration`：扫描完整性、单次解码、历史回填与失败处理。
- `DedupCore/persistence`：本地模型、同步消息、MySQL schema、读写字段和迁移。
- `DedupCore/dedup`：候选召回、二筛、三筛、严格成组、报告证据和选择安全。
- `VideoScGUI`：配置入口、进度、规则解释和旧报告兼容。
- `DedupTests`：边界、迁移、召回正确性、取消、缓存和真实变体覆盖。

## 3. 排查步骤

1. 建立“设计条款 → 源码锚点 → 测试锚点”对照表。
2. 沿扫描、持久化、报告、选择四条数据链逐字段检查。
3. 检查候选算法的召回数学条件、去重方式、复杂度和资源上限。
4. 检查错误、取消、低质量、缺路径、解码失败和旧版本数据行为。
5. 检查 GUI 展示是否与冻结报告元数据一致。
6. 运行只读搜索和必要的现有自动测试；不执行真实数据库迁移和用户数据扫描。
7. 输出中文根因/偏差文档，按严重程度列出证据、触发条件、影响和最小修复方向。
