# 扫描窗口当前阶段与进度条实现计划

> 日期：2026-07-15  
> 状态：已按推荐范围执行完成

## 1. 目标

在“扫描任务”窗口中，把当前仅有的阶段文本升级为清晰、实时且不误导用户的进度区域：

1. 明确显示当前正在进行的阶段及阶段序号。
2. 在阶段名称下方显示一条占满可用宽度的 ImGui 进度条。
3. 有可靠总量的阶段显示真实百分比与“已处理/总数”。
4. 无法预先得到总量的阶段显示动画式不确定进度，并展示持续增长的实际计数。
5. 成功、取消、失败和中断状态使用不同文案，不把失败任务显示成 100% 完成。
6. 保留现有任务 ID、累计计数、磁盘明细、RocksDB/MySQL 状态和错误提示。

## 2. 当前实现盘点

### 2.1 GUI 当前状态

`VideoScGUI/VideoScApp.cpp` 的 `RenderScanWindow()` 当前只显示：

- `阶段：<阶段名称>`。
- 任务 ID。
- 发现、SHA-512 完成、媒体特征完成、失败/跳过等累计数量。
- RocksDB、MySQL、共享同步状态。
- 各物理盘线程、队列和读取量表格。

当前没有调用 `ImGui::ProgressBar()`，用户只能看到累计数字，无法直观看出当前阶段是否仍在推进、还剩多少。

### 2.2 可直接使用的阶段

`ProgressSnapshot::phase` 和 `ScanPhaseName()` 已覆盖：

- 发现文件。
- 计算 SHA-512。
- 提取媒体特征。
- 同步 MySQL。
- 完成、取消、失败、中断等终态。

扫描协调器实际执行的本地流水线为：

```text
发现文件 -> 计算 SHA-512 -> 提取媒体特征 -> 本地完成
                                                |
                                                +-> 等待后台 MySQL 队列清空 -> 同步完成
```

`Syncing` 枚举目前没有由 `ScanCoordinator` 直接写入；GUI 只在本地完成且队列清空后，把阶段改成 `CompletedSynchronized`。因此本次需要在 GUI 快照刷新时把“本地已完成但共享队列未清空”明确派生为“同步 MySQL”。

### 2.3 现有计数不能直接当作进度分母

不能使用 `hashed_files / discovered_files` 作为哈希进度，原因是：

- `discovered_files` 包含增量扫描中直接复用旧 SHA-512 的文件。
- `hashed_files` 只统计本轮重新哈希且成功的文件。
- 哈希失败文件记入 `failed_files`，不计入 `hashed_files`。

不能使用 `media_processed_files / discovered_files` 作为媒体进度，原因是：

- 只有图片和视频需要媒体特征。
- 相同 SHA-512 的媒体任务会按内容去重。
- 增量扫描可能只重新处理算法版本过期的媒体。

因此必须为哈希阶段和媒体阶段提供独立的“本阶段总数”和“本阶段已处理数”，不能用现有累计字段拼出虚假百分比。

## 3. 推荐展示方案

扫描窗口在现有分隔线后显示：

```text
当前阶段：2/4  计算 SHA-512                     任务 ID：123456
[██████████████--------------------------]  1,248 / 3,920（31.8%）

发现 8,615 | SHA-512 成功 1,230 | 媒体特征完成 0 | 失败/跳过 18
```

阶段序号按本地扫描及最终同步过程显示：

1. 发现文件。
2. 计算 SHA-512。
3. 提取媒体特征。
4. 同步 MySQL。

若某阶段没有任务，可以快速进入下一阶段，不人为停留。后台 MySQL 实际上可以提前并行消费队列，但只有本地扫描完成后，才把它作为用户当前需要等待的主阶段显示。

### 3.1 各阶段进度规则

| 当前阶段 | 进度条类型 | 进度条覆盖文字 |
| --- | --- | --- |
| 空闲 | 0% 静态 | `等待开始扫描` |
| 发现文件 | 不确定动画 | `正在发现文件，已发现 N 个` |
| 计算 SHA-512 | 确定进度 | `已处理 P / T（X%）` |
| 提取媒体特征 | 确定进度 | `已处理 P / T（X%）` |
| 同步 MySQL | 不确定动画 | `正在同步，已同步 N 条操作` |
| 本地与 MySQL 同步完成 | 100% 静态 | `扫描与同步全部完成` |
| 本地完成但无需等待同步 | 100% 静态 | `本地扫描完成` |
| 已取消 | 保留最后一次可计算值 | `扫描已取消` |
| 失败 | 保留最后一次可计算值并使用错误色 | `扫描失败` |
| 已中断/已暂停 | 保留最后一次可计算值 | `扫描已中断，可从断点恢复` / `扫描已暂停` |

发现阶段在目录遍历结束前没有总文件数；MySQL 队列当前也只读取一条记录判断“空/非空”，没有完整待同步总量。两个阶段使用 ImGui 原生动画式不确定进度，避免伪造百分比。

### 3.2 百分比边界

- 分母为零时不执行除法。
- 阶段总量为零且阶段已正常结束时，按该阶段已跳过处理并立即进入下一阶段。
- 确定进度统一限制在 `[0.0, 1.0]`。
- “已处理”必须包含成功和失败的本阶段任务，保证失败文件不会让进度永久停在 99%。
- 取消、失败或中断后不强制填满进度条。
- 新 `scan_id` 出现时清空上一任务保留的 UI 进度，防止新任务继承旧任务显示。

## 4. 数据模型调整

### `DedupCore/orchestration/ProgressSnapshot.h`

为避免改变现有字段语义，增加独立只读快照字段：

- `hash_total_files`：本轮实际需要哈希的文件总数。
- `hash_processed_files`：哈希工作线程已经返回结果的任务数，包含成功、读取失败、超时和取消前已返回的任务。
- `media_total_files`：当前扫描 ID 下按 SHA-512 去重后的媒体任务总数。

继续保留：

- `hashed_files` 表示哈希成功数。
- `media_processed_files` 表示媒体任务已完成并提交结果的数量。
- `failed_files` 表示失败/跳过累计数。

不复用当前未实际填充的 `queued_compute_tasks` 作为总量，因为“当前排队数”会随着任务消费而下降，不是稳定分母。

## 5. 扫描协调器调整

### 5.1 哈希总量

在发现线程全部结束、`discovered_by_disk_` 内容稳定后，统计所有物理盘缓冲中的待哈希文件数，写入 `hash_total_files`，再进入哈希阶段。

`HandleHashCompleted()` 对每一个已经返回的哈希任务递增 `hash_processed_files`；现有成功和失败计数仍按原逻辑维护。这样进度为：

```text
hash_processed_files / hash_total_files
```

### 5.2 媒体总量

进入 `ProcessMediaPhase()` 后，先按当前扫描 ID 的 `media/<scan_id>/` 前缀做一次只计数迭代，得到按 SHA-512 去重后的 `media_total_files`；随后继续使用现有有界队列做第二次迭代和并行处理。

该方案：

- 不把全部任务加载进内存。
- 不改变媒体任务的去重规则。
- 不修改 FFmpeg 并发和取消逻辑。
- 代价是媒体阶段开始前增加一次 RocksDB 前缀顺序遍历，但相对实际媒体解码开销很小，且可换来稳定、不倒退的真实进度。

若计数迭代失败，沿用现有 RocksDB 致命任务错误处理，不显示不可信百分比。

### 5.3 同步阶段

不修改 `MySqlSyncService` 的同步算法和队列格式。GUI 刷新快照时按以下条件派生阶段：

```text
local_scan_complete && !shared_sync_complete -> Syncing
local_scan_complete &&  shared_sync_complete -> CompletedSynchronized
```

同步期间显示 `m_syncSnapshot.synchronized_operations` 的实际累计数量，但不显示伪造的总数或百分比。

## 6. GUI 调整

### `VideoScGUI/VideoScApp.cpp`

1. 增加内部进度展示模型/辅助函数，根据 `ProgressSnapshot` 生成：
   - 当前阶段序号和名称。

   - 进度条覆盖文字。
   - 成功、警告、失败对应的颜色。
2. 在 `RenderScanWindow()` 的阶段文本位置插入全宽 `ImGui::ProgressBar()`。
3. 发现和同步阶段使用当前 ImGui 已支持的负时间值动画模式，不新增自定义绘制控件。
4. 保留原有累计统计和磁盘表格，避免进度条替代诊断信息。
5. 在 `RefreshRuntimeSnapshots()` 中补充同步阶段的 UI 派生逻辑。

### `VideoScGUI/VideoScApp.h`

如需在取消、失败和中断后保留最后一帧进度，增加最小 UI 状态：

- 上一次扫描 ID。
- 上一次可计算的阶段进度值。

这些字段只服务展示，不写入配置文件或扫描检查点。

## 7. 不修改范围

- 不修改文件发现、SHA-512、dHash、FFmpeg 分析算法。
- 不修改磁盘调度策略、线程数和队列容量。
- 不修改 RocksDB 业务记录、MySQL schema 或同步消息格式。
- 不修改 `config.json` schema。
- 不把不同耗时特征的阶段硬拼成一个“总体百分比”。
- 默认不把“生成重复报告”和“永久删除”纳入本次扫描进度条，除非确认扫描窗口需要统一展示这两个独立任务。
