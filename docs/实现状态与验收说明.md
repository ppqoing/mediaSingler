## 2. 千万级 dHash 索引说明

图片 64 位 dHash 按配置的最大汉明距离 `n` 均分为 `n+1` 个连续段，并按段序号和值分别写入 RocksDB 桶。汉明距离不超过 `n` 时，至少有一段完全一致；桶命中后仍执行真实汉明距离复核。`n` 可在配置页面设置为 `0～15`，默认 `4`。

图片和视频相似组采用完整链接规则：新内容必须与候选组内全部不同 SHA-512 内容逐一通过真实相似判断，才允许加入该组，不再通过相似边的传递闭包合并。

随机数据下，候选量的近似量级由原单段方案的 `5N/8192` 降为约 `15N/2^21`。在 `N=10,000,000` 时，期望候选量从每项约 6100 降到约 70。索引键还包含完整视觉签名，相同 dHash 或相同六帧签名只保留一组索引项；报告生成时再把同签名的 SHA-512 内容线性并入代表节点，避免热门签名退化为二次方比较。

## 4. 尚需用户环境验收

1. 保留配置中的 MySQL 数据库，手动删除全部 VideoSc 表后点击“初始化数据库表”，确认 schema 3 和空的 `videosc_data_version` 表创建成功。
2. 启动扫描，确认首次数据版本缺失会清理旧 RocksDB 派生数据并进入 `rebuilding`，不会删除源媒体、配置或日志。
3. 等待扫描和 MySQL 队列归零，确认 RocksDB/MySQL 变为同 generation `ready`，随后自动精确报告与手动视觉报告均可生成。
4. 使用真实 MySQL 8 账号继续验证 TLS、已有库备份、断线重连和大批量补同步。
5. 使用约 10 块真实 HDD/SSD 验证每盘线程配置、HDD 排序收益、坏块超时和长时间稳定性。
6. 准备千万级真实或合成记录，记录候选数、RocksDB 容量、报告耗时和峰值内存。
7. 在测试副本目录人工走通 GUI 的“扫描 → 自动精确报告 → 手动相似报告 → 条件选择 → 删除选中文件”。
8. 核对执行日志、MySQL 路径映射和本地文件结果一致后，再用于正式媒体库。

## 5. 安全边界

- 程序不会自动开始扫描或删除源媒体；永久删除必须经过条件/手动选择和确认弹窗。
- 数据版本缺失、低于当前版本或无法证明两端一致时，运行时会自动清理固定 RocksDB Column Family 和固定 VideoSc MySQL 业务表；不会执行 `DROP DATABASE`，也不会触碰非 VideoSc 表。
- 删除不进入回收站。任何文件完整 SHA-512 不一致、读取失败或超时都会跳过。
- 相似组允许删除，但“视觉相似”不等于字节相同，必须先在 GUI 查看缩略图。
- 正常同版本运行中，删除文件只移除 MySQL 路径-SHA512 映射；受控数据版本升级会重建全部派生业务表。
