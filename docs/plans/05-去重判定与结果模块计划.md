# 去重判定、选择与永久删除模块计划

## 1. 模块目标

从 MySQL 共享数据生成 SHA-512 精确重复组和图片/视频 dHash 内容相似组，提供可解释的比较证据、质量排序、路径与磁盘筛选、每组保留一个的硬约束，并在用户二次确认后永久删除选中文件。文件操作和数据库清理分阶段执行，删除及移动写入文件日志。

## 2. 重复组类型

### 2.1 SHA-512 精确重复组

- `file_path_sha512` 中同一个 SHA-512 关联至少两个不同物理文件路径。
- 自动从 MySQL 读取并生成，无需用户手动计算。
- 所有成员字节相同，大小必然相同。
- 硬链接路径不计为独立可释放副本。

### 2.2 图片内容相似组

- 用户手动触发内容相似扫描。
- `sha512_file_data` 中图片 dHash 的汉明距离严格 `< 5`。
- 不做旋转归一化。
- SHA-512 可以不同，属于内容相似而非精确相同。

### 2.3 视频内容相似组

- 用户手动触发内容相似扫描。
- 两个视频均不是静态画面视频。
- 时长绝对差不超过 2 秒。
- 六组对应 dHash 汉明距离的平均值严格 `< 5`。
- UI 展示每个视频的完整 2×3 拼图。

### 2.4 静态画面视频

单个视频六帧的 15 组两两汉明距离平均值 `< 5` 时标记 `StaticVisual`。该视频不进入视频内容相似组，只参与 SHA-512 精确重复组。

## 3. 自动精确分组

1. MySQL 为 `file_path_sha512(sha512, path_id)` 建立覆盖索引；RocksDB 同步维护 SHA-512 到路径引用的增量计数。
2. 日常增量只重建引用数发生变化的 SHA-512；全量重建按二进制 SHA-512 有序流式扫描，一遍聚合，复杂度接近 O(N)，内存只保留当前组。
3. 排除已确认缺失、待删除和文件身份无效的路径。
4. 合并硬链接到同一内容实体，避免重复计数。
5. 路径数量或物理内容数量满足条件时创建/更新精确组。
6. 增量路径加入或删除后只重建受影响 SHA-512 的组。

精确组可使用 SHA-512 和算法版本派生稳定业务键，数据库仍使用内部主键管理关系。

## 4. 手动内容相似匹配

### 4.1 图片索引

不能对千万级图片做全量 O(N²) 比较。把 64 位 dHash 确定性拆成 5 个 12/13 位片段，按 `(algorithm_version, segment_position, segment_value)` 在 RocksDB 建持久化倒排表。若两个 dHash 距离不超过 4，最多有 4 位不同，因此至少一个片段完全相同；查询五个桶的并集不会漏掉合法候选。候选仍必须用真实 `ComputeHammingDistance` 复核。

完全相同或高度集中的大桶不生成所有候选对，而是先按完整 dHash 聚合，再用代表成员与其他桶候选比较，避免空白图等常见摘要造成组合爆炸。

### 4.2 视频候选

先按 2 秒时长桶及相邻桶建立候选范围，最终复核绝对差不超过 2 秒，并排除 `StaticVisual`。视频平均距离 `< 5` 等价于六帧距离和不超过 29；若六组对应帧距离都至少为 5，则总和至少 30，因此合法候选必有至少一个帧位距离不超过 4。对六个帧位分别建立与图片相同的 15 个两段联合键，共 90 个 RocksDB 候选键；取并集、限制时长后再计算完整六帧距离和。

候选索引不会改变最终规则：只有六帧都有效、时长差不超过 2 秒、距离和不超过 29 时才成立。

### 4.3 千万级索引流水线

- 索引只针对 `sha512_file_data` 中的唯一内容，不为相同 SHA-512 的每条路径重复建索引。
- RocksDB 使用独立 `exact_index`、`image_hash_index` 和 `video_hash_index` Column Family。
- posting list 使用压缩、分片和有界批量合并，不把单个热门桶全部放入内存。
- MySQL 通过主键键集分页流入索引，不使用大 offset 分页。
- 新增或算法版本变化的内容增量更新索引；路径新增只更新 SHA-512 引用，不重建内容索引。
- 候选对以 `(min_content_id, max_content_id)` 规范化，并通过 RocksDB 批次去重或规范桶归属防止重复计算。
- 匹配工作按索引前缀分片到全局计算线程，检查点写 RocksDB，可暂停、恢复和重试。

### 4.3 分组语义

dHash 相似关系不保证传递。若 A 与 B 相似、B 与 C 相似但 A 与 C 不相似，不应默默把三者当成完全互相相似。结果模型需要保存边证据，并选择以下可解释策略之一：

- 以代表文件为中心的星型组；或
- 连通分量组，但 UI 明确展示每对距离。

第一版建议代表文件星型组，避免链式相似把差异过大的文件合并。代表文件由当前保留策略选择，改变代表时重新验证组成员。

## 5. 结果证据

### 5.1 组信息

- 组类型、算法和版本。
- 成员数、在线数、硬链接数。
- 精确组 SHA-512，或内容相似组代表文件和距离。
- 总占用、预计释放空间。
- 当前保留策略和强制保留成员。
- MySQL 同步版本与结果生成时间。

### 5.2 成员信息

- 路径、物理磁盘、文件大小和最后修改时间。
- 图片或视频分辨率、视频码率。
- SHA-512、dHash 距离和媒体状态。
- 图片缩略图路径或视频 2×3 拼图路径。
- 是否硬链接、是否选中删除、保留优先级和错误。

## 6. 可释放空间

### 6.1 精确组

按独立物理内容计算：所有可删除独立副本大小之和。硬链接路径不增加可释放空间。

### 6.2 内容相似组

成员内容不同，显示“选中项预计释放空间”，不能称为字节重复的安全空间。用户仍可永久删除，但 UI 必须显示相似证据和缩略图。

## 7. 批量选择条件

所有条件只改变选择状态，不立即删除：

### 7.1 删除较大版本

适用于 dHash 内容相似组。选择文件体积较大的成员并保留体积最小成员。SHA-512 精确组大小相同，该条件禁用或提示无效果。

### 7.2 删除低质量版本

- 视频质量从高到低：分辨率像素数 → 视频码率 → 文件大小。
- 图片质量从高到低：分辨率像素数 → 文件大小。
- 选择低质量成员，保留最高质量成员。

### 7.3 指定物理磁盘

- 选择指定磁盘中的重复成员。
- 若组在其他磁盘有未选成员，可选中该磁盘全部成员。
- 若组全部位于指定磁盘，自动保留一个。

### 7.4 保留策略选项

- 最新：最后修改时间最大。
- 最旧：最后修改时间最小。
- 最小：文件大小最小。
- 最大：文件大小最大。
- 路径优先级：GUI 扫描路径列表顺序靠前者优先。

用户可以批量条件后手工勾选或取消。每次选择变化都立即重新执行“每组至少保留一个”约束。

## 8. 强制保留一个

删除计划冻结前对每组执行：

1. 统计在线、可操作成员。
2. 如果没有选中全部成员，保留现状。
3. 如果全部被选中，按当前保留策略撤销一个成员。
4. 如果保留策略无法决胜，按路径优先级和稳定路径键决胜。
5. GUI 显示“系统强制保留”原因，用户不能再次选中最后成员。

该规则同时适用于 SHA-512 精确组和 dHash 内容相似组。

## 9. 永久删除流程

### 9.1 预览

- 独立“永久删除选中文件”按钮。
- 展示组数、文件数、路径、物理盘、预计空间、保留成员和内容相似缩略图。
- 明确提示不可恢复。
- 用户二次确认后冻结批次。

### 9.2 前置检查

- 日志目录可写且滚动日志成功打开。
- RocksDB 可写，能够记录每项状态和 MySQL tombstone。
- 路径仍指向预期文件身份。
- SHA-512 精确组可重新检查摘要或文件身份。
- dHash 组检查媒体记录版本和当前 SHA-512 未变化。
- 每组仍至少保留一个在线成员。

### 9.3 文件删除与映射删除

1. 逐个调用永久文件删除，不进入回收站或隔离目录。
2. 每项立即写文件日志和 RocksDB 结果。
3. 失败项保留本地文件和 MySQL 路径映射。
4. 全部本地操作完成后，把成功项组成 MySQL 批量删除请求。
5. 直接删除 `file_path_sha512` 对应映射行。
6. 不删除 `sha512_file_data`。
7. MySQL 失败时 tombstone 保留在 RocksDB，后台补删。
8. 批次完成后刷新受影响重复组。

### 9.4 日志

日志至少记录时间、批次 ID、动作、路径、SHA-512、组类型、选择条件、保留成员、Win32 错误和 MySQL 映射状态。日志写文件，不建立删除审计数据库表。日志滚动、保留天数和目录可配置。

## 10. 移动文件处理

程序扫描到移动后的新路径时重新计算 SHA-512：

- 命中已有 `sha512_file_data`：新增路径映射并复用媒体数据和拼图。
- 未命中：按新内容提取媒体数据。
- 旧路径在扫描中标记缺失；只有确认由程序永久删除时才直接批量删除映射行。

## 11. 建议文件布局

```text
DedupCore/matching/
├─ ExactDuplicateMatcher.h/.cpp
├─ ImageSimilarityIndex.h/.cpp
├─ VideoSimilarityMatcher.h/.cpp
├─ DuplicateGroup.h
└─ ResultQueryService.h/.cpp

DedupCore/deletion/
├─ RetentionPolicy.h/.cpp
├─ SelectionRule.h/.cpp
├─ DeletePlan.h
├─ PermanentDeleteExecutor.h/.cpp
└─ OperationFileLogger.h/.cpp
```

匹配器不直接删除文件；删除执行器只接受已冻结且通过保留约束的计划。

## 12. 实施任务

### M1：精确分组

- MySQL SHA-512 聚合、硬链接和增量刷新。

### M2：图片相似索引

- 6 段的 15 个两段联合 RocksDB 键、完整签名热门桶压缩、距离 0 至 4 候选和真实距离复核。

### M3：视频匹配

- 2 秒时长桶、六帧 30 个分段键、静态排除、距离和不超过 29 和星型分组。

### M3.1：千万级断点

- MySQL 键集分页、RocksDB posting list、候选对去重、索引版本和增量更新。

### M4：结果与缩略图证据

- 分页结果、拼图/图片路径、质量和距离字段。

### M5：选择与保留

- 较大、低质量、指定磁盘、最新、最旧、最小、最大和路径优先级。
- 每组强制保留一个。

### M6：永久删除

- 预览、二次确认、前置检查、文件日志、批量映射删除和 RocksDB tombstone。

## 13. 验收标准

1. SHA-512 相同路径自动形成精确组。
2. 图片只有距离 `< 5` 才形成内容相似边。
3. 视频只有时长差不超过 2 秒且六组平均距离 `< 5` 才匹配。
4. 静态画面视频不进入视频内容相似组。
5. 选择规则和删除按钮完全分离。
6. 所有条件组合后，每个组仍至少保留一个成员。
7. dHash 组在永久删除预览中显示缩略图或 2×3 拼图。
8. 删除失败文件的路径映射不会被删除。
9. 本地删除成功后，映射只在批次末尾删除。
10. `sha512_file_data` 不因路径删除而删除。
11. MySQL 失败时 RocksDB 可恢复映射删除。
12. 所有删除和移动动作进入文件日志，数据库没有删除审计表。
13. 千万级 SHA-512 精确分组使用单遍有序聚合，不执行路径两两比较。
14. 图片/视频内容相似使用无漏候选的分段索引，并只对候选计算真实距离。
15. 中断后从 RocksDB 索引检查点继续，不重建全部已完成索引。

## 14. 风险与对策

| 风险 | 对策 |
| --- | --- |
| dHash 误判导致永久删除 | 拼图/缩略图、距离证据、独立选择、二次确认、强制保留一个 |
| 相似关系链式扩散 | 代表文件星型组并保存逐对证据 |
| 删除后 MySQL 失败 | RocksDB tombstone 和幂等批量补删 |
| 日志丢失 | 日志不可写时禁止启动删除批次 |
| 文件扫描后被替换 | 删除前重新核对身份和 SHA-512 版本 |
| 热门 dHash 桶造成候选爆炸 | 完整 dHash 先聚合、posting list 分片、代表成员和有界批次 |
| 千万级候选重复计算 | 规范内容 ID 对、规范桶归属和 RocksDB 去重检查点 |
