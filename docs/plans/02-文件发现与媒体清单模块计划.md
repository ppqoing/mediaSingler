# 文件发现与媒体清单模块计划

## 1. 模块目标

把 GUI 配置的多个扫描路径转换为可断点、可增量、可按物理磁盘调度的全部文件清单。该模块发现所有普通文件，不以扩展名、大小或媒体类型决定是否计算 SHA-512；媒体分类只决定后续是否提取图片或视频内容数据。

## 2. 当前能力与问题

原 GUI 的 `EverythingFileListQuery` 支持 Everything SDK 动态加载、进程启动、分页查询、长路径和取消，但其结果仍全量进入内存。生产扫描已选择 Win32 原生流式枚举；Everything 只保留在可选诊断工具中。原型问题包括：

1. 每个根目录结果全部进入 `std::vector`，随后复制到集合和排序向量。
2. 直接生成 `disk_<n>.txt`，无法支持断点、同步或媒体服务器数据。
3. 重叠根目录只按路径字符串去重，不能可靠识别同一文件和硬链接。
4. 扫描路径配置与查询线程都位于 GUI 项目。
5. 没有把每次发现变更写入 RocksDB。

## 3. 模块边界

### 3.1 输入

- 不可变的 `ScanOptions` 和 `scan_id`。
- 有序扫描根目录列表。
- 原生目录枚举器和取消状态。
- 存储拓扑解析服务。
- RocksDB 批次接收器。

### 3.2 输出

每个 `DiscoveredFile` 至少包含：

- 原始路径、规范化路径和所属根目录顺序。
- 卷 GUID、文件 ID、物理磁盘调度键。
- 文件大小、扩展名、创建时间、修改时间和属性。
- 初步媒体类型。
- 在线状态、发现时间和错误。
- 是否需要重新计算 SHA-512。

### 3.3 不负责

- 不在发现线程计算 SHA-512 或 dHash。
- 不访问 MySQL 热路径。
- 不判定重复或执行删除。
- 不保存全量结果到 GUI。

## 4. 扫描路径管理

### 4.1 GUI 契约

- 支持浏览选择、手工添加、移除和拖动排序。
- 列表顺序既是扫描顺序提示，也是删除时的路径保留优先级。
- 任务开始时复制为不可变快照。
- 运行中修改路径只影响下一任务。
- 移除路径不清除历史；RocksDB 中标记 `OutOfScope`，MySQL 路径映射保留，除非本地文件被永久删除。

### 4.2 根目录规范化

1. 内部使用 UTF-16 Windows 路径。
2. 展开绝对路径、规范分隔符和卷挂载点。
3. 生成不受展示大小写影响的比较键。
4. 使用卷 GUID 和 `FILE_ID_INFO` 获取文件身份。
5. 默认不跟随 reparse point，避免循环和越出根目录。
6. 支持长路径，不用固定 `MAX_PATH` 保存结果。

## 5. Win32 原生流式发现设计

### 5.1 每根目录独立发现线程

每个扫描根目录使用独立发现线程和 `FindFirstFileExW/FindNextFileW` 流式枚举。发现结果立即规范化并提交对应物理盘有界队列；队列满时形成背压。默认不跟随 reparse point，避免目录环和越出配置根目录。

### 5.2 流式投递

- 每个路径立即规范化、读取基础元数据并投递 RocksDB 与 SHA-512 调度器。
- 不保留整个根目录的路径向量。
- 不建立全目录路径数组；持久化和各物理盘队列容量形成固定内存上界。
- 队列满时发现线程等待，形成背压。
- 批次成功持久化后才推进本地发现检查点。

### 5.3 所有文件入清单

- 目录不进入 SHA-512 队列。
- 所有可访问普通文件进入清单，不限图片、视频、音频或其他格式。
- 扩展名和文件头只用于媒体分类。
- 零字节文件同样计算 SHA-512。
- 系统/隐藏文件是否包含由配置决定，但默认规则必须写入任务快照。

## 6. 文件身份、重叠路径和硬链接

- 同一规范化路径在重叠根目录中只形成一个当前路径记录，但保留其根目录归属关系。
- 同一文件 ID 的多个硬链接保留多条路径映射，共享一次 SHA-512 计算结果。
- 释放空间计算不把硬链接路径算成独立数据副本。
- 无法取得文件 ID 时退化为规范化路径，并记录退化状态。
- 路径被移动后作为新路径发现，仍重新计算 SHA-512；若 SHA-512 已存在 MySQL 内容表，只新增映射。

## 7. 增量扫描规则

### 7.1 可复用

同一路径或稳定文件身份的大小、修改时间和文件 ID 都未变化，并且 RocksDB 中已有成功 SHA-512 时，可以复用摘要，不重新读取文件。该行为属于增量扫描，不违背“所有文件都计算 SHA-512”，因为每个内容版本至少完整计算一次。

### 7.2 必须重算

- 新路径。
- 文件 ID、大小或修改时间变化。
- 上次哈希中断、失败或算法版本变化。
- 用户要求强制重扫。
- RocksDB 记录缺失或校验失败。

### 7.3 缺失和离线

- 上次存在、本次未发现：标记 `Missing`。
- 物理盘不在线：标记 `Offline`，不把全部路径误判为已删除。
- 根目录从配置移除：标记 `OutOfScope`。
- 这些状态不删除 `sha512_file_data`。

## 8. RocksDB 键设计建议

- `scan/{scan_id}/root/{root_id}`：根目录快照和进度。
- `scan/{scan_id}/path/{normalized_path}`：本次发现记录。
- `file/{volume_id}/{file_id}`：本地文件身份和最近有效 SHA-512。
- `work/sha/{storage_target}/{record_id}`：待哈希任务。
- `sync/mysql/{sequence}`：待同步操作。

键和值需要版本字段和校验，不能把 C++ 内存结构直接持久化。

## 9. 建议文件布局

```text
DedupCore/discovery/
└─ NativeFileDiscovery.h/.cpp
```

现有 `VideoScGUI/EverythingFileListQuery.*` 仅用于可选诊断窗口，不参与生产扫描、断点或去重流程。

## 10. 实施任务

### M1：数据契约

- 定义根目录、文件身份、发现状态和 RocksDB 序列化版本。
- 明确 UTF-16 内部路径边界。

### M2：原生流式发现

- 使用 Win32 长路径兼容枚举和大目录优化标志。
- 逐文件输出并依靠物理盘有界队列背压，删除生产流程的全量结果容器和文本中间文件。

### M3：路径和身份

- 实现规范化、卷 GUID、文件 ID、硬链接和重叠根目录处理。
- 实现 GUI 路径顺序到根目录优先级的持久化。

### M4：增量清单

- 与 RocksDB 联调新建、复用、重算、缺失、离线和移出范围。
- 生成每物理盘 SHA-512 工作项。

### M5：诊断枚举

- Everything 原型只保留在诊断工具，不影响生产扫描。
- 生产枚举保持流式、取消和错误契约。

## 11. 验收标准

1. 所有可访问普通文件都进入 SHA-512 清单。
2. 百万级路径不会在发现模块形成全量多份内存。
3. 重叠根目录、硬链接和 Unicode 长路径正确处理。
4. 新路径即使内容可能存在，也会进入 SHA-512 计算。
5. 未变化文件可从 RocksDB 增量复用。
6. `Missing`、`Offline` 和 `OutOfScope` 不会删除共享内容数据。
7. 取消后停止枚举新路径，已提交记录保持一致。
8. 生产扫描不依赖 Everything 服务或其索引时效。
9. 输出可直接驱动物理盘调度，不依赖 `disk_<n>.txt`。

## 12. 风险与对策

| 风险 | 对策 |
| --- | --- |
| 枚举期间文件变化 | 哈希前后重新读取文件身份、大小和修改时间，变化则不接受摘要 |
| 文件量巨大 | 分页、RocksDB 批次、有界队列 |
| 盘离线被误判删除 | 目标级在线状态和 `Offline` 标记 |
| 路径移动导致内容重算 | 新路径计算 SHA-512，命中 MySQL 内容数据后复用媒体结果 |
| 重叠路径重复扫描 | 规范化键、文件身份和根目录关系分离 |
