## 1. 已确认目标

基于现有 `VideoSc.sln` 实现 Windows x64 去重工具，面向约 10 块 HDD/SSD 混合磁盘。不同物理磁盘并行读取，同一物理磁盘按用户配置控制读取线程数，并额外限制全局计算线程数。

配置页面集中管理每物理盘读取线程、全局计算线程、FFmpeg 单任务线程、读取块和队列容量、HDD 排序窗口、坏块重试、读取与数据库超时、MySQL 同步批量、缩略图生成尺寸及缓存预算。任务开始时冻结配置快照，运行中修改只影响下一任务；dHash 阈值、视频六帧采样和 2 秒时长规则保持固定。

配置页面的持久化来源固定为安装目录下的 `config.json`，采用 UTF-8 JSON 和模式版本。MySQL 密码使用当前 Windows 用户范围的 DPAPI 加密，Base64 后写入 JSON，不保存明文。保存使用同目录临时文件、刷盘和原子替换，并保留上一份有效备份；安装目录不可写时明确报错，不静默改存其他目录。

机械硬盘采用顺序大块读取，并在 NTFS 上按文件物理区间做有界电梯排序，减少磁头往返。遇到坏块、CRC、I/O 设备错误或长时间无读取进展时，对当前块有限重试并缩小块重试；仍失败则把文件标记为不可读、写入日志并跳过，不暂停整个物理盘，也不生成部分 SHA-512。

单次任务可能处理千万级路径。SHA-512 使用二进制索引和单遍有序聚合；图片 dHash 切成 6 段并建立全部 15 个“两段联合”候选键；视频先按时长分桶，再对六个帧位各建立 15 个候选键，共 90 个。汉明距离不超过 4 时，六段中至少有两段完全一致，因此不漏召回。完整视觉签名作为索引键后缀，相同 dHash 只占一组索引键，避免热门哈希形成巨型倒排链。索引持久化在 RocksDB，并按新增内容增量维护，禁止千万级全量两两比较。

所有扫描文件都必须计算完整 SHA-512，用于字节级精确去重。图片使用 dHash 汉明距离 `< 5` 识别内容相似；视频在 `t/7` 至 `6t/7` 生成六帧，比较两个时长差不超过 2 秒的视频，六组对应 dHash 的平均汉明距离 `< 5` 时判为内容相似。音频只参与 SHA-512 精确去重，不进行声纹或内容相似去重。

每个视频只生成一个 2 行 × 3 列的拼图缩略图文件，数据库保存其路径。单个视频六帧形成的 15 组两两汉明距离平均值 `< 5` 时标记为静态画面视频；静态画面视频不参与视频 dHash 相似去重，只使用 SHA-512。

扫描期状态、断点、队列和待同步数据保存在一个持久化 RocksDB 中；通过 `scan_id` 区分任务。共享媒体数据保存在 MySQL 8.0+。MySQL 不可用时扫描继续，待连接恢复后从 RocksDB 批量补同步。

工具支持精确重复和 dHash 相似组的批量选择、永久删除。选择条件与“删除选中文件”是分开的操作；每个重复组必须至少保留一个文件。删除本地文件成功后，在批次末尾删除 MySQL 的路径-SHA512 映射行；SHA512-文件数据表只增量，不因路径删除而删除。删除、移动和数据库操作写入滚动文件日志，不写删除审计表。

## 3. 当前实现状态

1. `VideoSc` 已实现 Overlapped 流式 SHA-512、图片 dHash、视频六帧精确采样、静态画面判定和单张 2×3 拼图。
2. `DiskInfo` 与 `DedupCore` 已实现物理盘识别、HDD 物理区间有界电梯排序、每盘读取线程和全局计算并发。
3. `DedupCore` 已实现原生流式发现、断点与增量扫描、RocksDB 热表、MySQL 持久化队列和自动补同步。
4. 精确报告从 MySQL 流式读取后分页写入 RocksDB；相似报告由 GUI 手动触发，使用六段两两联合索引、真实距离复核和 RocksDB 并查集。
5. `VideoScGUI` 已接入扫描监控、断点恢复、报告分页与懒加载缩略图、条件选择、永久删除确认和删除结果汇总。
6. 永久删除会复核保留副本和待删文件完整 SHA-512，先写删除意图，文件操作全部结束后批量删除路径映射并唤醒 MySQL 同步。
7. “初始化数据库表”是唯一数据库结构按钮；已有表时先调用随包部署的 `mysqldump.exe` 生成备份，再执行幂等建表。
8. 坏块、读取错误与超时写入 `scan-errors.jsonl`；删除操作写入 `operations.jsonl`，两者均不写审计数据库。

Release 构建和 28 个核心自动化测试已经通过。尚未完成的环境验收包括：使用用户真实 MySQL 8 实例验证建表、备份与断线补同步；使用约 10 块真实 HDD/SSD 进行长时间吞吐、坏盘和千万级索引压测；人工完成 GUI 全流程视觉验收。没有这些环境证据前，不把相关项标记为已实盘验证。

## 4. 推荐项目结构

`DedupCore` 静态库承载发现、调度、RocksDB、MySQL 同步、分组、日志和删除编排。`VideoSc` 保留 FFmpeg/哈希能力，`DiskInfo` 提供存储拓扑，`VideoScGUI` 只负责命令、配置、进度和审阅。`DedupTests` 覆盖算法、并发、持久化、增量扫描、报告和删除安全。

## 5. 实施纪律

- 所有文件必须计算完整 SHA-512，不再用文件大小或快速采样指纹跳过文件。
- SHA-512 相同自动形成精确重复组。
- dHash 相似计算由用户在 GUI 手动触发。
- 视频相似比较仅在时长差不超过 2 秒时执行。
- 静态画面视频只参与 SHA-512 去重。
- 永久删除前先批量选择并预览，执行时每组强制保留一个。
- 路径映射只有在本地永久删除成功后才在批次末尾删除。
- RocksDB 和 MySQL 同步状态必须可见，不能把“本地完成”误报成“共享数据已完成”。
- 所有删除和移动动作写文件日志；日志失败时禁止开始新的永久删除批次。
- 坏块、读取超时和不可读偏移写文件日志；只跳过相关文件，不熔断或自动暂停整个机械盘。
- 千万级精确和内容哈希比较必须通过持久化索引、键集分页和流式聚合完成。
